# 个股持续研究与爆点追踪

## Purpose

本 skill 负责回答一个动态问题：

```text
这条新信息，是否让原有投资逻辑发生升级、降级、高潮或证伪？
```

它不是完整个股深研的替代品。完整深研由 `$supply-chain-agentic-research`、`$chassis-growth-agentic-research`、`$industry-chain-agentic-research` 或对应子 skill 完成；本 skill 接在这些研究之后，用于持续跟踪“证据是否落位”。

本 skill 不因股价涨幅大而自动否定成长股，也不因股价强而自动看多。高不是风险本身，势能衰减才是风险；低不是安全本身，逻辑没有增量也可能是陷阱。

核心原则：

```text
先固定基准假设 -> 再判断新信息的增量 -> 映射到财报/估值/市场行为 -> 更新阶段和分数
```

## When to use

适合用户要求：

- 跟踪某只股票、某条产业链主线或一组标的的最新爆点。
- 判断一条公告、财报、业绩预告、投资者关系记录、客户发布会、订单消息是否构成强验证。
- 判断一只股票从“有逻辑”到“被强证据验证”的节点。
- 判断一只成长股涨幅较大后，是否仍有新的起爆势能，而不是简单畏高或追高。
- 判断某条主线是否已经进入逻辑高潮、拥挤交易或证伪阶段。
- 把已有深研结论转成后续持续跟踪清单。
- 用户明确说“不怕多用资源”“可以启动 subagent”“深度核验”“不要碎片化输出”时，使用 Formal Tracker，并由主控整合多 agent 发现。

不适合单独使用：

- 用户要求从零写正式深度报告。应先调用完整深研类 skill。
- 用户只问静态基本面、财务拆解、产业链科普。应优先使用对应行业或成长股研究 skill。
- 没有任何公司或主线对象，且用户只泛泛询问方法论。可以简要解释框架，但不需要完整执行。

## Operating Modes

根据任务强度选择模式。

### Quick Event Check

用于判断单条新闻、公告、财报或股价异动。可以直接在聊天中输出，不强制建文件。

必须回答：

- 新信息是什么，日期和来源是什么。
- 它解决了哪个核心分歧。
- 它是旧共识重复，还是真正增量。
- 证据等级、爆点等级、阶段变化。
- 后续最需要跟踪的 1 到 3 个证据。

### Formal Tracker

用于持续跟踪一只股票或一条主线。建立或更新目录：

```text
research_artifacts/<股票代码或公司简称>/
  <标的>_baseline_card.md
  <标的>_catalyst_log.md
  <标的>_stage_scorecard.md
  <标的>_watchlist.md
  <标的>_agent_findings.md
  <标的>_integrated_update.md
```

Formal Tracker 的全部跟踪文件必须写入上述目录；不得把投研文件写到任务根目录、`artifacts/` 或其他临时目录。

若已有完整深研文件，先读取其结论、证据缺口和证伪点，再建立跟踪文件，避免重复深研。

### Recurring Monitor

当用户明确要求“每天/每周/持续盯/有变化提醒”时，先定义跟踪对象、核心分歧、证据源和输出格式；若当前环境支持自动化工具，再创建相应 automation。不要把一次性跟踪误建成定时任务。

## Agent Policy

本 skill 可以使用多 agent / subagent 做资源密集型取证，但必须保持主控完整性。

### When to use agents

在用户明确允许使用 subagent、要求深度跟踪、复盘爆点、比较多个证据源、判断重大财报/公告/客户事件，或任务明显需要并行核实时，优先启动多角色取证。资源不是主要约束，质量和完整判断优先。

Quick Event Check 可以不启动 agent；但如果事件涉及重大客户、财报、技术路线或估值锚切换，也可以升级为多 agent 快速核验。

如果当前环境无法启动真实 subagent，用分阶段文件化方式模拟同样角色，不得因此跳过反方审查或市场行为检查。

### Required roles for formal tracking

正式跟踪至少覆盖 5 个角色。可以由真实 subagent 完成，也可以由主控分阶段完成。

| 角色 | 任务 |
| --- | --- |
| 原始证据 agent | 查公告、财报、业绩预告、投资者关系记录、交易所披露，确认日期、原文口径和事实边界 |
| 客户与产业 agent | 查客户 capex、路线图、平台发布、技术路线、同行财报和产业链交叉验证 |
| 财报映射 agent | 判断事件如何传导到收入、毛利率、现金流、存货、预付款、在建工程和利润中枢 |
| 市场与估值 agent | 检查股价、成交、估值锚、机构预期、板块扩散和是否出现右侧确认或高潮钝化 |
| 反方审查 agent | 找替代解释、证据缺口、证伪信号、拥挤风险和“把故事当事实”的跳跃 |

### Main controller duties

主控不能把多个 agent 的结果拼贴成碎片。主控必须完成二次合成：

- 合并重复证据，去掉低质量噪音。
- 解决不同 agent 之间的冲突；如果冲突无法解决，明确列为“待确认分歧”。
- 把所有证据串成一条完整链路：外部需求/客户信号 -> 公司订单/产品/产能 -> 财报科目 -> 利润中枢 -> 估值锚 -> 市场行为。
- 给出唯一的阶段判断和爆点等级；不要让不同角色各自给一个互相打架的结论。
- 最终只输出整合后的投资研究判断，agent 发现可以作为附录或证据表，不得喧宾夺主。

正式跟踪时，`<标的>_agent_findings.md` 只保存分工发现，`<标的>_integrated_update.md` 才是面向用户的统一结论来源。

## Source Rules

动态跟踪必须重视时效性和来源等级。

- 涉及“最新、最近、今天、昨晚、刚发布、Q1/Q2/Q3、业绩预告、股价、市值、订单、价格、客户资本开支”等信息时，必须查最新来源并写明具体日期。
- 优先使用公司公告、交易所披露、财报、投资者关系记录、业绩说明会、客户官方新闻稿、客户财报和电话会。
- A股数据可结合 `$tushare`、`$wencai-query`、`$akshare`；但市场口径和问财结果不能替代公告事实。
- 如果引用媒体、卖方研报或产业链传闻，必须标明其可靠性低于官方和客户证据。
- 必须区分事实、推断、假设、市场口径和价格行为。

## Evidence Levels

每条信息都先按来源和验证强度分级。

| 等级 | 类型 | 判断 |
| --- | --- | --- |
| 0 | 无来源传闻、论坛小作文、情绪扩散 | 只能作为线索，不能作为判断依据 |
| 1 | 行业泛泛利好 | 有背景意义，但不是公司级验证 |
| 2 | 卖方研报、媒体复盘、市场解读 | 可提供框架，需回到原始证据 |
| 3 | 同行、供应商、产业链交叉验证 | 说明主线可能是行业级变化 |
| 4 | 公司官方验证 | 公告、财报、业绩预告、投资者关系、订单、扩产、产品进展 |
| 5 | 客户或产业巨头验证 | 下游客户 capex、路线图、点名、认证、平台发布、采购指引 |
| 6 | 连续财务兑现 | 两个季度以上收入、利润、毛利率、现金流、订单/扩产共同验证 |

## Catalyst Levels

每次更新都要给出爆点等级。

| 等级 | 含义 | 典型信号 |
| --- | --- | --- |
| -2 | 核心逻辑受损 | 客户 capex 下修、订单取消、技术路线排除、利润或现金流严重背离 |
| -1 | 预警信号 | 毛利率下滑、库存异常、价格战、扩产无订单支撑、利好不涨 |
| 0 | 噪音 | 没有新事实，或只是重复旧共识 |
| 1 | 弱增量 | 行业景气或情绪改善，但未传导到公司 |
| 2 | 公司级小增量 | 新产品、扩产、订单、财报有边际进展，但未解决核心分歧 |
| 3 | 分歧部分解决 | 客户认证、订单加速、出货爬坡、财报初步兑现 |
| 4 | 强验证 | 客户/订单/财报/现金流共同指向逻辑升级 |
| 5 | 重估爆点 | 核心分歧大幅消失，估值锚切换，或高位出现新一轮盈利预期/产业身份/市场认知上修，并出现右侧市场确认 |

## Six-Dimension Update

对重要信息，用 0 到 5 分更新六个维度。不要机械打分；分数必须有证据解释。

| 维度 | 核心问题 |
| --- | --- |
| 需求强度 | 需求是短期波动，还是 1 到 3 年可见的放量周期？ |
| 供给约束 | 行业是否存在产能、良率、认证、技术、供应链瓶颈？ |
| 公司卡位 | 公司是核心供应商、关键平台、弹性配套，还是边缘受益？ |
| 财报传导 | 逻辑是否进入收入、利润率、现金流、存货、预付款、在建工程？ |
| 估值锚变化 | 公司身份是否从旧标签切换到更高估值的新标签？ |
| 市场认知差 | 市场是否仍有分歧，股价是否已经完全定价？ |

## Stage Taxonomy

必须判断当前处于哪个阶段。

| 阶段 | 特征 |
| --- | --- |
| 预期型 | 产业逻辑存在，但订单和财报尚未充分验证 |
| 半兑现型 | 已出现客户、订单或财报初步验证，但仍有关键分歧 |
| 兑现型 | 收入、利润、现金流或订单连续验证，逻辑稳定性提高 |
| 右侧重估型 | 估值锚切换，股价和成交确认市场开始重新定价 |
| 二次起爆型 | 前期已有明显涨幅，但新证据继续抬升盈利预期、产业身份或估值锚，市场出现新一轮右侧确认 |
| 高潮型 | 逻辑广泛共识化，利好钝化，后排乱涨，估值按乐观情景定价 |
| 证伪型 | 关键假设被客户、订单、价格、技术路线或财报破坏 |

## High-Position Re-acceleration

成长股涨幅较大后，不要用“高位”直接替代研究判断。必须判断新的势能是否足以带来二次起爆。

高位二次起爆必须回答：

- 新证据是新变量，还是旧逻辑重复？例如新客户、新代际产品、新技术路线确认、新价格周期、新产能瓶颈、新利润率台阶。
- 它解决的是剩余核心分歧，还是只是在强化已经充分定价的共识？
- 盈利预期是否继续上修，而且上修速度是否能解释股价涨幅？重点看收入指引、订单可见度、毛利率、EPS、现金流和远期利润中枢。
- 估值锚是否再次变化？例如从普通供应商变成关键平台，从周期股变成 AI 基建瓶颈资产，从单产品公司变成多产品平台。
- 市场行为是否健康？龙头是否继续强于后排，利好后是否仍能上行，成交放大是否伴随承接，而不是单纯剧烈换手。
- 如果新势能落空，原逻辑是否仍能支撑估值，还是会出现快速回撤？

判断口径：

```text
高位不可怕，缺少新增盈利预期、估值锚或认知差才可怕。
继续上涨不可直接等于泡沫，利好钝化和证据不再增量才更接近高潮。
```

## Climax Checks

不要只会找强，也要识别过热。

出现以下组合时，要提示逻辑可能从“加强”进入“高潮”：

- 利好不涨，财报超预期但股价钝化。
- 研报、媒体、社交平台标题高度一致，分歧明显减少。
- 板块后排和低质量标的补涨。
- 当前估值已经按极乐观利润情景定价。
- 成交剧烈放大，但龙头抗跌性下降。
- 公司继续兑现，但市场上修幅度开始小于股价涨幅。

## Financial Mapping

把产业新闻翻译成财务报表信号。

| 新闻/事件 | 应观察的财务科目 |
| --- | --- |
| 客户 capex 上修 | 收入、订单、存货、在建工程、资本开支 |
| 订单加急或排产饱满 | 预付款、存货、应收、产能利用率、收入确认 |
| 高端产品放量 | 毛利率、ASP、产品结构、净利率 |
| 供给紧张或涨价 | 毛利率、合同价格、库存、客户锁单 |
| 扩产 | 在建工程、固定资产、折旧、现金流、产能爬坡 |
| 技术路线确认 | 研发投入、客户认证、新产品收入、资本开支方向 |
| 行业价格战 | 毛利率、存货跌价、应收质量、费用率 |

如果新闻很热但财报科目长期没有响应，必须降低逻辑强度。

## Non-Negotiables

- 每个强判断必须写清楚：证据 -> 推理 -> 替代解释 -> 判断强度。
- 允许多 agent 分工，但最终答案必须是主控整合后的单一研究判断，不能把 agent 小结并列堆叠给用户。
- 正式跟踪必须覆盖原始证据、客户产业、财报映射、市场估值、反方审查五类视角；缺少任何一类都要说明缺口。
- 不要因为涨幅大就自动判定“高位危险”；必须比较新势能、盈利预期上修、估值锚变化和市场是否仍有认知差。
- 也不要因为“强者恒强”就自动看多；如果新信息只是旧共识重复、利好钝化或估值已经按极乐观情景定价，必须提示高潮风险。
- 不要把“行业利好”直接跳成“公司受益”，也不要把“扩产”直接等同于“订单”。
- 不要把“订单”直接等同于“利润”，必须检查收入确认、毛利率、费用、现金流。
- 不要只看股价强弱。价格是反馈，不是证据本身。
- 不要只说“利好”或“利空”，必须说明相对市场原预期的增量。
- 对高波动、杠杆 ETF、高相关仓位，要明确相关风险和回撤风险。
- 输出必须可执行：最后给出下一步最该跟踪的证据，而不是只给观点。
