# Agentic 编排细则

## 总控职责

总控不是亲自写完所有分析，而是像投研项目经理一样组织研究。

总控必须完成：

- 提出主线假设。
- 判断框架适用性。
- 建立中间产物目录。
- 调度数据工具和研究角色。
- 处理不同角色之间的冲突。
- 决定最终报告里哪些模块主导结论，哪些只是支撑或风险约束。

## 角色分工与内置模块映射

本 skill 内置供应链平台型成长股六个模块的问题清单，不依赖独立子 skill。详细检查项见 [module-checklists.md](module-checklists.md)。

| 原框架模块 | 主要负责角色 | 作用 |
| --- | --- | --- |
| 产业周期 | 产业周期与客户 Capex agent | 判断下游需求、客户 capex、平台迭代和周期持续性 |
| 供应链地位 | 供应链地位与壁垒 agent | 判断位置、认证、交付壁垒、替代性和议价权 |
| 订单出货 | 订单出货与经营质量 agent | 判断订单、排产、出货、财报确认和增长质量 |
| 承接动作 | 订单出货与经营质量 agent | 判断扩产、融资、设备、量产、海外基地和研发投入 |
| 利润桥 | 利润中枢与估值 agent | 把订单和故事转成未来利润中枢 |
| 远期估值 | 利润中枢与估值 agent、反方审查 agent | 判断当前市值是否透支未来利润 |

主控可以把多个模块合并给同一个 agent，但不能省略任何模块。若某个模块证据不足，应写“证据不足”，而不是从报告中消失。

正式研究必须触发 5 个研究角色：产业周期与客户 Capex、供应链地位与壁垒、订单出货与经营质量、利润中枢与估值、反方审查。真实 subagent 是首选执行载体；当前环境、宿主规则和工具能力都允许时，必须用真实 subagent。当前环境、宿主规则或工具能力不允许时，必须用阶段文件模拟这些角色。区别只在执行载体，不在研究步骤是否存在。

### 产业周期与客户 Capex agent

任务边界：

- 下游周期和客户 capex。
- 产品平台迭代、价值量、复杂度、ASP。
- 周期阶段和供给价格风险。

不能做：

- 不要把行业 TAM 直接写成公司订单。
- 不要替代供应链地位、利润桥和估值判断。

### 供应链地位与壁垒 agent

任务边界：

- 产业链角色。
- 客户认证、合作深度、复杂产品量产。
- 规模交付、全球产能、供应链组织能力。
- 可替代性、议价权、客户集中风险。

不能做：

- 不要把“大客户供应商”直接等同于平台型公司。
- 不要因为公司不是最高技术定义者就直接否定供应链壁垒。

### 订单出货与经营质量 agent

任务边界：

- 订单、排产、出货、量产、收入确认。
- 强/中/弱信号分级。
- 毛利率、净利率、应收、存货、合同负债、经营现金流。
- 承接动作：扩产、融资、设备、海外基地、研发投入。

不能做：

- 不要把扩产当作订单。
- 不要把订单直接当作利润。
- 不要把收入增长直接当作高质量增长。

### 利润中枢与估值 agent

任务边界：

- 旧平台利润、新周期新增利润、扩张成本。
- 利润桥的保守、中性、乐观情景。
- 当前市值反推 2025、2026E、2027E、2028E PE。
- 估值是否低估、合理、偏贵或透支。

不能做：

- 不要把季度利润年化当作精确预测。
- 不要只给 PE 表，不解释当前市值隐含的年份和利润情景。

### 反方审查 agent

任务边界：

- 只挑错，不负责把报告写漂亮。
- 检查证据链跳步、估值误导、数据口径混用、风险遗漏。
- 检查普通投资人是否能理解。

输出文件：`<标的简称或代码>_skeptic_review.md`

输出结构：

```text

```

## 无真实 Subagent 时的模拟方式

如果当前环境、宿主规则或工具能力不能启动真实 subagent，不要退回一次性写作。必须按以下方式模拟：

1. 先写 `<标的简称或代码>_cycle_capex.md`，只处理产业周期。
2. 再读取该文件，写 `<标的简称或代码>_position_moat.md`，只处理供应链地位。
3. 再读取前两个文件，写 `<标的简称或代码>_orders_shipments_quality.md`，只处理订单出货和经营质量。
4. 再读取前三个文件，写 `<标的简称或代码>_profit_valuation.md`，只处理利润桥和估值。
5. 再读取前四个文件，写 `<标的简称或代码>_skeptic_review.md`，只挑错。
6. 最后读取所有文件，写提纲和终稿。

关键是每一轮都有独立文件，而不是把所有思考留在上下文里。

如果某一阶段信息不足，仍然要生成对应文件，并写明：

- 已查到的证据。
- 缺少的数据。
- 哪些判断只能降级为“证据不足”或“正在验证”。
- 下一步需要补什么信息。

## 冲突处理

当不同角色给出冲突判断时，总控必须显式处理：

- 周期 agent 认为下游需求强，但地位 agent 认为公司只是普通供应商：降级为“行业景气强，但公司重估弹性有限”。
- 地位 agent 认为公司壁垒强，但订单 agent 找不到订单出货放量证据：降级为“地位可能成立，但利润兑现待验证”。
- 订单 agent 认为出货放量，但利润中枢与估值 agent 发现毛利率或现金流恶化：不能写成“高质量增长”，要解释质量风险。
- 利润中枢与估值 agent 认为业绩高增，但反方 agent 提醒当前估值已经反映后年乐观情景：结论不能写“便宜”，应写“兑现要求高”。
- 公告和市场口径冲突：公告优先，市场口径作为交易情绪或同类比较。

## 最终整合

终稿不是拼接五份中间稿。总控必须重新排序：

- 先解释公司在产业链里做什么。
- 再解释下游周期为什么重要。
- 再解释公司为什么可能发生身份切换。
- 再解释订单如何落到收入、利润和现金流。
- 最后讨论利润平台、估值和证伪。

结论语言要体现条件性：

```text
这家公司不是因为当期 PE 低而有吸引力，而是因为产业周期、供应链地位和订单出货显示利润平台可能上移；但当前市值是否合理，取决于未来两到三年利润桥能否兑现，以及估值是否已经提前反映乐观情景。
```
