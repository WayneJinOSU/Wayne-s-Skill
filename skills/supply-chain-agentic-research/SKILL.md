# Agentic 供应链平台型成长股研究

## Overview

这是一个独立主控 skill，直接编排供应链平台型成长股研究的核心模块，并通过多角色研究、中间文件、市场变量扩表、利润天花板模型、终审 QA 和投资人写作控制输出质量。`final_report` 是本 skill 的主终点；它负责把产业链、市场真正交易的变量、上行动能、订单、经营质量、利润中枢和天花板讲清楚。正式估值由 `$growth-stock-valuation` 接手；本 skill 只准备估值接力输入，`final_report` 不写目标价、目标市值、PE/PEG/SOTP 或买卖建议。

核心判断仍然是：

```text
产业周期够大 + 市场变量足够锐利 + 技术路线到价值量讲透 + 原材料/价格/产能传导清楚 + 供应链位置关键 + 订单出货领先 + 第二曲线成型 + 利润平台换台阶 + 证伪路径明确
```

本 skill 的目标不是生成核心摘要，也不是只做证据分层备忘录，而是通过多角色研究、中间文件、进攻型变量建模、终审 QA 和投资人写作，把供应链平台股的逻辑讲完整：下游周期为什么成立，市场真正交易的变量是什么，技术迭代如何抬高单机/单套价值量，公司为什么不是普通代工，订单如何变收入，收入如何变利润，利润天花板如何被抬高，哪些指标会证伪主线。

## When to use

适合用户要求：

- 深度分析一只处在 AI 算力、数据中心、新能源、机器人、先进封装、半导体设备/材料等大产业周期中的供应链平台型公司。
- 判断公司是否发生身份切换、订单出货放量、第二曲线成型、利润中枢上移和天花板抬升，并把证据、推理、市场口径和证伪点讲清楚。
- 不只要核心结论，还要把产业周期、市场变量、技术路线到价值量、原材料传导、产能升级、供应链地位、订单出货、第二曲线和利润桥讲清楚。
- 需要用 subagent / 多角色 / 多阶段研究避免一次性摘要化输出。

不适合：

- 银行、地产、保险、纯消费品牌、纯资源周期、纯软件互联网、早期创新药。
- 无收入利润底盘、只有概念没有订单和产能承接的题材股。
- 用户只要“一句话看法”。如果用户要求“快速判断”“先粗筛”，可以使用轻量模式，但必须说明未完成完整 agentic 流程。

## Non-negotiables

正式研究必须同时满足以下要求：

1. 完整覆盖供应链平台研究的 6 个主攻模块、1 个终审 QA 和 1 条横向链路：
   - 市场交易变量主线
   - 架构/技术路线与天花板
   - 原材料价格链
   - 产能升级与第二曲线
   - 订单出货与经营验证
   - 利润天花板/利润桥
   - 终审 QA：反方观点和证伪检查
   - 横向链路：市场变量与技术路线到价值量链路
   产业周期、供应链地位、承接动作是嵌入检查项，不再默认作为平权 agent 或独立章节：周期嵌入市场变量和架构路线，地位嵌入架构路线和订单经营验证，承接动作嵌入产能升级和订单经营验证。模块文件放在 `modules/` 目录下；技术路线横向链路使用 [references/technology-route-value-chain.md](references/technology-route-value-chain.md)。进入正式估值阶段时，调用 `$growth-stock-valuation`；估值结论不写入 `final_report`。
2. 不允许直接输出终稿。必须先形成中间研究产物。
3. 每个核心投资判断必须写清楚：

```text
证据 -> 推理 -> 反证/替代解释 -> 判断强度
```

4. 必须区分：
   - 公告或财报事实
   - 客户 capex / 行业数据
   - 订单、排产、出货或海关口径
   - 结构化行情和财务数据
   - 市场口径
   - 合理推断
   - 情景假设
5. 最终报告必须面向一般投资人，解释术语、铺设推理台阶，不能只堆概念、数据和结论。
6. 对市场正在交易但公告尚未充分验证的变量，必须主动搜集、扩表和建模。市场口径、券商假设、产业链线索和交易热词不能只放在证据台账、脚注或终审 QA 里；它们要进入主线、天花板、价格传导、敏感性和证伪路径。
7. 证据分层服务于判断，不替代判断。核心变量段落的结尾必须落到“成立时如何增厚利润/抬高天花板，不成立时逻辑降级成什么”。每个高重要变量都必须写出：

```text
变量增强时，收入、ASP、毛利率、费用、良率、现金流会如何变化。
变量减弱时，逻辑会降级成什么：周期修复、补库涨价、低端放量、扩产先行、利润被成本吞噬，还是主线证伪。
```

8. 技术路线升级型供应链公司必须单独写“价格与利润传导”。不得只写“高端化”“结构升级”“国产替代”。必须拆清上一代和新一代产品的价格/价值量差异、单机/单柜/单车/单套价值量变化、高端产品毛利率溢价、产品占比提升敏感性、原材料与良率扣减。若只有市场口径，必须标为情景假设或压力测试，但仍要用于解释变量弹性。
9. 必须写“天花板和势能”。公告事实负责确定下限和事实边界；券商、产业链、专家纪要和交易热词负责识别弹性变量。对高重要市场口径，要给出底线/基准/上行三层情景或至少给出可调整的驱动公式：产品代际/ASP/毛利率/份额/产能利用率/良率/少数股东权益如何共同改变利润中枢。不得在 `final_report` 中写目标价、目标市值、PE、PEG、SOTP 或买卖建议。

## Research Posture

正式报告要同时有进攻性和边界感。先把市场正在交易的上行模型讲完整，再用证据等级和终审 QA 约束它；不要先把变量降噪成低弹性结论。

- 公告事实确定底线：收入、利润、现金流、产能、项目进展、官方披露产品和客户边界。
- 市场口径打开弹性：产品代际、客户份额、订单排产、ASP、毛利率、单机价值量、关键原料、良率、供应商清单。
- 情景假设负责天花板：用可调整的数字或区间展示“如果市场口径成立，利润中枢怎样抬高；如果不成立，回到哪一档”。
- 终审 QA 负责证伪：集中写关键证伪点，不要在每一段反复弱化主线。

写作上优先使用可判断表达：

- 如果只是普通产品放量，收入会涨但利润弹性有限；如果高端产品规模量产，同样出货面积会带来更高 ASP 和毛利。
- 如果原材料涨价能顺利转嫁，是利润放大器；如果需求降温或客户压价，就是利润挤压器。
- 如果扩产先于客户导入，会先体现为折旧、费用和爬坡成本；如果客户认证和产能利用率同步提升，才会变成收入和利润上限。
- 如果平台地位只体现在综合份额，它支撑规模和交付；如果进入最高端细分客户清单，它才支撑更高毛利率和更强客户粘性。
- 如果新一代架构/平台要求材料等级跃迁，同一台设备/同一柜/同一车/同一套系统的价值量会被重新计算；若公司进入核心材料清单，天花板上移，若只停留在普通产品供应，逻辑回到周期修复。

表格只能做索引或 QA。凡表格中出现的核心变量，正文必须至少用一段解释其需求来源、公司受益环节、财务传导、变量增强/减弱时的结论变化。不得用表格替代正文推理。

### Market Variable Writing Rule

对所有高重要市场变量，终稿必须采用以下写法：

```text
市场口径说什么：A、B、C。
证据等级是什么：公告事实、券商口径、产业链口径、专家纪要、媒体/自媒体或弱线索。
为什么市场交易它：它抬高 TAM、单机价值量、ASP、毛利率、客户粘性、份额，还是利润年份切换。
如果成立：收入、ASP、毛利率、份额、客户替换成本、良率、现金流或归母利润会怎样变。
如果不成立：逻辑降级成什么，是普通周期修复、补库涨价、低端放量、扩产先行、成本挤压，还是主线证伪。
如何验证：公司公告、客户认证、订单金额、分产品收入/毛利率、产销量、现金流、应收存货、合同负债、资本开支或客户 capex 中哪一项能验证。
```

## Agent Policy

正式研究必须触发多角色研究，但角色权重要向主线集中。6 个主攻 agent 负责推进上行逻辑；1 个终审 QA 只在终稿前集中审查，不提前压扁主线。

真实 subagent 是首选执行载体。当前环境、宿主规则和 subagent 工具能力都允许时，必须启动 subagent 分工完成以下 6 个主攻角色和 1 个终审 QA：

- 市场变量主线 agent：负责主动搜集券商、产业链、专家纪要、交易热词、竞品、客户 capex 和终端路线，形成市场交易变量扩表。周期信息只回答“为什么新增需求会穿透到公司产品”，不得单独扩写成行业景气综述。
- 架构/技术路线与天花板 agent：负责终端平台/架构路线、产品代际、单机/单柜/单车/单套价值量、利润天花板抬升。AI 方向要优先查 GB200/GB300/Rubin/Feynman、M7/M8/M9/M10、800G/1.6T、HBM/先进封装等。供应链地位只保留能解释 ASP、毛利率、份额、认证门槛和客户替换成本的部分。
- 原材料价格链 agent：负责关键原料成本结构、价格涨跌、供应紧缺、同行调价、成本转嫁、库存和毛利率传导。
- 产能升级与第二曲线 agent：负责基地地图、扩产/投产/爬坡、产品定位、海外基地、控股/参股公司、第二利润曲线和协同关系。承接动作中的扩产、设备、海外基地、原料锁定归入本 agent。
- 订单出货与经营验证 agent：负责订单、排产、出货、收入确认、毛利率、应收、存货、现金流、客户认证、供应链地位和承接动作验证。地位不单独写“龙头/平台”话术，只回答是否能变成订单稳定性、ASP、毛利率、份额、现金流和客户替换成本。
- 利润天花板/利润桥 agent：负责存量利润底盘、新周期新增利润、利润天花板、利润斜率、扩张成本、市场变量的利润传导、并表/权益法/少数股东处理。
- 终审 QA：终稿前集中寻找证据缺口、替代解释、市场变量误用、周期见顶风险、把扩产当订单、把收入当利润、把市场口径当事实等问题。它只负责校准和证伪，不作为平权章节抢主线。

每个主攻 agent 都必须输出本角色识别到的“市场变量交付块”和“上行情景交付块”。只要涉及产品代际、客户份额、ASP、毛利率、订单/排产、良率、核心客户认证、关键原料、产能利用率、单机价值量或第二曲线，即使证据弱，也要写清：市场口径说什么、证据等级、若成立如何改变收入/毛利率/份额/现金流/利润天花板、若不成立逻辑降级成什么、应在终稿哪个章节保留。

如果当前环境、宿主规则或工具能力不允许真实 subagent，必须用“分阶段文件化”的方式模拟上述 6 个主攻角色和 1 个终审 QA：每个阶段单独产出文件，下一阶段读取文件后再继续，不要只在上下文里滚动压缩。不要把“无法启动真实 subagent”当作跳过角色的理由。

任何角色无法获取足够信息时，仍要输出市场正在交易的假设、缺少的数据、合理的情景边界和验证路径，不能跳过该角色。

## Workflow

### Step 1: 建立研究目录

在当前任务目录下建立中间产物目录：

```text
research_artifacts/<股票代码或公司简称>/
  <标的简称或代码>_question.md
  <标的简称或代码>_evidence_index.md
  <标的简称或代码>_market_variables_map.md
  <标的简称或代码>_architecture_value_ceiling.md
  <标的简称或代码>_raw_material_price_chain.md
  <标的简称或代码>_capacity_second_curve.md
  <标的简称或代码>_orders_business_validation.md
  <标的简称或代码>_profit_bridge.md
  <标的简称或代码>_valuation_handoff.md（估值接力输入；终稿仍不写正式估值）
  <标的简称或代码>_valuation_deepdive.md（若调用 `$growth-stock-valuation`）
  <标的简称或代码>_valuation_scorecard.md（若调用 `$growth-stock-valuation`）
  <标的简称或代码>_skeptic_review.md
  <标的简称或代码>_report_outline.md
  <标的简称或代码>_final_report.md
```

### Step 2: 主线假设与适用性分诊

先写入 `<标的简称或代码>_question.md`：

- 市场正在交易什么：身份切换、订单放量、利润中枢上移、技术路线迭代、市场变量兑现或证伪。
- 需要验证的 3 个核心问题。
- 先写“进攻型主线假设”：如果市场变量成立，利润天花板如何抬升；如果不成立，逻辑降级到哪一档。
- 公司是否适用供应链平台型框架；如不适用，停止完整流程并说明更合适框架。

### Step 3: 证据收集

写入 `<标的简称或代码>_evidence_index.md`，按可获得性覆盖：

- 公告原文：年报、季报、业绩预告/快报、订单、合同、募投、定增、可转债、问询回复、业绩说明会。
- 客户和行业资料：客户财报、资本开支指引、电话会、产品平台路线、行业出货、权威机构或协会数据。
- `$tushare`：股价、市值、PE/PB、财务三表、财务指标、现金流、历史行情。
- `$wencai-query`：题材概念、市场排名、同行筛选、阶段涨幅、事件性条件。
- 海关、出货、排产和产业链跟踪数据；若不是官方来源，必须说明可靠性和口径限制。
- 市场增量情报：券商研报核心假设、产业链调研、专家问答、公司官网产品参数、竞品募资书、交易热词、终端技术路线、客户/竞品财报、供应链扩产和原材料价格。此类资料不能直接等同事实，但必须用于识别市场真正交易的边际变量。不得只读本地公告就进入多角色研究。

证据台账必须识别“市场变量”和“利润天花板变量”，但表格只能作为索引或 QA，不能替代正文推理。证据台账中的高重要市场变量必须被显式传递到 `<标的简称或代码>_market_variables_map.md`、`<标的简称或代码>_report_outline.md` 和 `<标的简称或代码>_final_report.md`；若终稿不保留，必须在市场变量覆盖 QA 中说明删除原因。对每个高重要市场变量，必须用段落写清：

```text
变量：市场正在交易什么。
为什么会带来新增需求：下游变化如何改变用量、价值量、规格、认证门槛或供需关系。
公司在哪个环节受益：公司是材料、零部件、设备、制造、系统、渠道，还是控股/参股公司受益。
财务传导路径：影响量、价、毛利率、份额、费用、折旧、良率、少数股东权益、投资收益中的哪几项。
证据属性：公告事实、客户/行业一手资料、市场/券商口径、情景假设或弱线索。
变量增强时：收入、ASP、毛利率、订单稳定性、现金流或利润中枢会怎样变化。
变量减弱时：逻辑会降级成什么，是普通周期修复、补库涨价、低端放量、扩产先行、成本挤压，还是主线证伪。
```

### Step 3.5: 市场交易变量扩表

在分发底座 agent 前，先写入 `<标的简称或代码>_market_variables_map.md`。这是正式研究的方向盘，必须回答：

- 市场当前最关心的 5-12 个变量是什么，包括终端架构/平台、产品代际、客户份额、ASP、毛利率、订单排产、关键原料、产能瓶颈、第二曲线和利润天花板。
- 每个变量来自哪里：公告事实、行业/客户一手资料、券商口径、产业链口径、专家纪要、媒体/自媒体、合理推断或情景假设。
- 每个变量为什么抬高天花板或改变利润斜率。
- 若变量成立，收入、ASP、毛利率、份额、客户替换成本、良率、现金流、少数股东权益或投资收益如何变化。
- 若变量不成立，逻辑降级成哪一档。
- 哪个 agent 负责继续深挖，终稿哪个章节必须保留。

若标的处于技术路线快速迭代的供应链环节，必须建立通用链路：下游平台迭代 -> 性能瓶颈变化 -> 零部件/材料规格升级 -> 单机价值量、ASP、毛利率或用量变化 -> 供应商认证、份额、良率、产能约束 -> 订单、收入、利润和现金流兑现。详细方法见 [references/technology-route-value-chain.md](references/technology-route-value-chain.md)。

若存在明确产品代际或材料等级升级，例如 M6/M7 -> M8/M9、400G -> 800G/1.6T、风冷 -> 液冷、普通连接器 -> 高速连接器、普通电源 -> 高功率密度电源，证据台账必须保留价格/价值量口径：

```text
上一代产品价格、价值量或毛利率口径。
新一代产品价格、价值量或毛利率口径。
价差或毛利率差来自哪里：性能、认证、良率、稀缺、客户粘性，还是短期供需。
公司能拿到多少：公司事实、市场口径、情景假设必须分开。
对收入和毛利的压力测试：占比提升 5/10/20 个百分点，或单机价值量变化，对收入/毛利/现金流有什么影响。
扣减项：原材料、良率爬坡、折旧、费用、少数股东损益、投资收益重复计算。
```

证据优先级和数据路径见 [references/data-path.md](references/data-path.md)。

### Step 6: 面向普通投资人的终稿

读取全部中间文件后，再写 `<标的简称或代码>_final_report.md`。这是主终点之一，也是最终面向用户的核心交付。

终稿结构优先使用下列顺序；若内容重叠，可以合并章节，但不能省略对应逻辑。终稿不单列估值章节，不放目标价、目标市值、PE/PEG/SOTP 表，也不写“估值透支、隐含利润、赔率、定价”等估值判断；估值材料统一放在 `valuation_handoff`。

1. 一句话结论：公司是预期型、半兑现型、兑现型供应链平台成长，还是仍停留在主题映射/周期修复。
2. 先讲人话：这家公司在产业链里到底做什么。
3. 市场交易变量与主线：市场为什么重新关注它，正在押哪些订单、技术路线、价格和利润变量。
4. 架构/技术路线与价值量：下游平台迭代如何改变性能瓶颈、材料/零部件规格、单机/单柜/单套价值量和供应商门槛。
5. 天花板抬升模型：传统周期高点、当前利润底盘、新路线利润区间分别是什么，差额来自产品代际、ASP、份额、产能、第二曲线还是费用率摊薄。
6. 原材料价格链与传导：关键原料价格、供需、转嫁机制、库存重估和毛利率留存。
7. 产能升级与供给约束：普通产能、高端产能、瓶颈产能、海外产能和第二曲线产能分别如何影响收入上限与爬坡成本。
8. 订单出货与经营验证：客户 capex 是否穿透到订单，供应链地位是否变成订单稳定性、ASP、毛利率、份额、现金流和客户替换成本。
9. 利润中枢和敏感性：写利润斜率、关键变量、底线/基准/上行情景和证伪条件；完整利润桥作为扩展项。
10. 终审 QA、证伪点和最终判断。

终稿写作必须执行以下 QA：

```text

2. 若这些词出现在核心变量段落，必须改写成“变量增强/减弱时结论如何变化”。
3. 搜索“高端化、国产替代、结构升级、平台型”等抽象词。
4. 若这些词没有伴随价格、价值量、毛利率、良率、份额、现金流或客户替换成本解释，必须重写。
5. 搜索证据台账和提纲中的高重要市场变量名称，例如产品代际、客户名/代称、ASP、毛利率、份额、订单、排产、良率、关键原料等。若这些词没有进入 final_report 正文，必须补写或在市场变量覆盖 QA 中解释合并/删除原因。
6. 搜索“估值、目标价、目标市值、PE、PEG、SOTP、隐含利润、赔率、定价”等词。
7. 若出现在 final_report，必须删除或移至 valuation_handoff。
```

## Output Discipline

最终回复用户时，除非用户要求只看终稿，否则要说明：

- 新生成了哪些中间文件。
- 哪些 agent / 阶段完成了哪些任务。
- 6 个主攻模块、1 个终审 QA 和 1 条技术路线横向链路如何被覆盖，最好给出覆盖表。
- `final_report` 是否已经完成。
- 终稿文件路径。
