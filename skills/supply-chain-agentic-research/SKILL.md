# Agentic 供应链平台型成长股研究

## Overview

这是一个独立主控 skill，直接编排供应链平台型成长股研究的核心模块，并通过多角色研究、中间文件、市场变量扩表、行业周期、竞争客户链、产品代际矩阵、利润天花板模型、两轮 QA 和投资人写作控制输出质量。`final_report` 是本 skill 的主终点；它负责把产业链、市场真正交易的变量、上行动能、订单、经营质量、利润中枢和天花板讲清楚，并形成对标优秀深度报告的正式研究稿，不是研究摘要、审计底稿或证据清单。正式估值由 `$growth-stock-valuation` 接手；本 skill 只准备估值接力输入，`final_report` 不写目标价、目标市值、PE/PEG/SOTP 或买卖建议。

核心判断仍然是：

```text
产业周期够大 + 市场变量足够锐利 + 技术路线到价值量讲透 + 原材料/价格/产能传导清楚 + 供应链位置关键 + 订单出货领先 + 第二曲线成型 + 利润平台换台阶 + 证伪路径明确
```

本 skill 的目标不是生成核心摘要，也不是只做证据分层备忘录，而是通过多角色研究、中间文件、进攻型变量建模、两轮 QA 和投资人写作，产出能与优秀券商深度、产业链深度和投资人主笔报告同台比较的研究稿：行业周期在哪里，技术迭代如何改变价值量，竞争对手和客户认证如何决定份额，公司产能如何承接订单，订单如何变收入，收入如何变归母利润，利润天花板如何被抬高，哪些季度指标会让主线继续升级或被证伪。

## When to use

适合用户要求：

- 深度分析一只处在 AI 算力、数据中心、新能源、机器人、先进封装、半导体设备/材料等大产业周期中的供应链平台型公司。
- 判断公司是否发生身份切换、订单出货放量、第二曲线成型、利润中枢上移和天花板抬升，并把证据、推理、市场口径和证伪点讲清楚。
- 不只要核心结论，还要把产业周期、市场变量、技术路线到价值量、原材料传导、产能升级、供应链地位、订单出货、第二曲线和利润桥讲清楚。
- 需要用 subagent / 多角色 / 多阶段研究避免一次性摘要化输出。

不适合：

- 银行、地产、保险、纯消费品牌、纯资源周期、纯软件互联网、早期创新药。
- 无收入利润底盘、只有概念没有订单和产能承接的题材股。
- 用户只要“一句话看法”。如果用户要求“快速判断”“先粗筛”，可以使用轻量模式，但必须说明未完成完整 agentic 流程。

## Non-negotiables

正式研究必须同时满足以下要求：

1. 完整覆盖供应链平台研究的 6 个纵向主攻模块、3 个横向增强模块、2 轮 QA 和 1 条技术路线横向链路：
   - 市场交易变量主线
   - 架构/技术路线与天花板
   - 原材料价格链
   - 产能升级与第二曲线
   - 订单出货与经营验证
   - 利润天花板/利润桥
   - 行业周期与供需格局：市场空间、周期位置、供需缺口、价格周期
   - 竞争格局与客户认证链：同行、客户、份额、认证、替代风险
   - 跟踪体系与警报信号：季度跟踪日历、红黄绿灯、敏感性和证伪触发
   - 中期结构 QA：检查缺章、缺表、缺模型、缺客户链和过度摘要化
   - 终审事实 QA：反方观点、事实边界和证伪检查
   - 横向链路：市场变量与技术路线到价值量链路
   产业周期、供应链地位、承接动作既要有横向模块校准，也要嵌入主线：周期不能停在行业综述，必须穿透到市场变量、架构路线和订单；地位不能停在龙头话术，必须穿透到客户认证、份额、ASP 和毛利率；承接动作必须回到产能升级和订单经营验证。模块文件放在 `modules/` 目录下；技术路线横向链路使用 [references/technology-route-value-chain.md](references/technology-route-value-chain.md)。进入正式估值阶段时，调用 `$growth-stock-valuation`；估值结论不写入 `final_report`。
2. 不允许直接输出终稿。必须先形成中间研究产物。
3. 每个核心投资判断必须写清楚：

```text
证据 -> 推理 -> 反证/替代解释 -> 判断强度
```

4. 必须区分：
   - 公告或财报事实
   - 客户 capex / 行业数据
   - 订单、排产、出货或海关口径
   - 结构化行情和财务数据
   - 市场口径
   - 合理推断
   - 情景假设
5. 最终报告必须面向一般投资人，解释术语、铺设推理台阶，不能只堆概念、数据和结论。
6. 对市场正在交易但公告尚未充分验证的变量，必须主动搜集、扩表和建模。市场口径、券商假设、产业链线索和交易热词不能只放在证据台账、脚注或终审 QA 里；它们要进入主线、天花板、价格传导、敏感性和证伪路径。
7. 证据分层服务于判断，不替代判断。核心变量段落的结尾必须落到“成立时如何增厚利润/抬高天花板，不成立时逻辑降级成什么”。每个高重要变量都必须写出：

```text
变量增强时，收入、ASP、毛利率、费用、良率、现金流会如何变化。
变量减弱时，逻辑会降级成什么：周期修复、补库涨价、低端放量、扩产先行、利润被成本吞噬，还是主线证伪。
```

8. 技术路线升级型供应链公司必须单独写“价格与利润传导”。不得只写“高端化”“结构升级”“国产替代”。必须拆清上一代和新一代产品的价格/价值量差异、单机/单柜/单车/单套价值量变化、高端产品毛利率溢价、产品占比提升敏感性、原材料与良率扣减。若只有市场口径，必须标为情景假设或压力测试，但仍要用于解释变量弹性。
9. 必须写“天花板和势能”。公告事实负责确定下限和事实边界；券商、产业链、专家纪要和交易热词负责识别弹性变量。对高重要市场口径，要给出底线/基准/上行三层情景或至少给出可调整的驱动公式：产品代际/ASP/毛利率/份额/产能利用率/良率/少数股东权益如何共同改变利润中枢。不得在 `final_report` 中写目标价、目标市值、PE、PEG、SOTP 或买卖建议。
10. `final_report` 的第一目标是把投资逻辑、证据边界、利润传导和跟踪证伪讲清楚，而不是为了显得正式而堆厚度，也不是为了刚好通过最低闸门而压缩成研究备忘录。默认正式报告应完整覆盖行业、技术、竞争、客户、产能、订单、利润模型和跟踪体系，通常以 10,000-14,000 中文字为宜。若公司变量简单、证据有限或用户明确要求精简，可压缩到 8,000-10,000 中文字，但必须在 `report_outline` 说明压缩原因；若公司存在 6 个以上高重要市场变量、多个产品代际/客户链/利润线或类似江波龙这类复杂平台型标的，通常应落在 12,000-16,000 中文字；若用户明确要求长篇深度版、券商深度版或需要对标 25-40 页报告，可以扩展到 15,000-25,000 中文字。若资料不足或逻辑本身较集中，不得为满足字数而重复铺陈，应合并章节并说明原因。
11. 终稿必须是全链条成稿，不得只按 6 个模块拼接。必须显式覆盖：行业空间与周期位置、技术路线与产品代际、竞争格局、客户/认证链、产能与供给约束、订单与经营验证、利润模型、跟踪日历和红色警报信号。
12. 技术升级型公司必须输出“产品代际矩阵”，至少包含：代际/产品、应用场景、关键性能指标、价格或价值量口径、毛利率口径、公司状态、竞争对手状态、证据等级、验证指标。缺少数字也要以“公告事实/市场口径/情景假设/缺口”标注，不能省略。
13. 利润桥必须从方向性升级为可计算模型。至少建立收入拆分、产量/出货量、ASP/价值量、产品占比、毛利率、费用率、折旧、良率、少数股东损益、投资收益、所得税、现金流和敏感性表。每个关键数字必须标注来源层级：公告事实、结构化数据、券商口径、产业链口径、自建假设或缺口。
14. 若用户给出参考报告、PDF、研报或样稿，必须先做“对标报告校准”：抽取其目录、章节密度、表格/模型类型、核心假设、证据边界和写作风格，再生成差距表；正式终稿必须达到同类深度，或说明哪些部分因证据不足而不采用。
15. `final_report` 可以写利润中枢、利润年份切换、市场正在交易的预期差和经营情景，但不写目标价、目标市值、PE/PEG/SOTP、买卖评级或明确投资建议；这些进入 `valuation_handoff` 或 `$growth-stock-valuation`。
16. 对 A 股供应链研究，搜索层必须高召回。优先组合使用公告网站、交易所、公司官网、客户/竞品公告、券商研报摘要、行业机构、结构化数据和普通 web search；进入证据收集前必须检查 `GEMINI_API_KEY`。若可用，必须运行 [scripts/gemini_google_search.py](scripts/gemini_google_search.py) 调用 Google Search 作为“线索发现器”；若不可用，必须在证据台账记录 `Gemini未运行原因`。Gemini/Google 搜索结果只能进入证据台账的“市场口径/线索”层，必须回到原始公告、研报 PDF、公司/客户资料或结构化数据复核后，才能支撑正文判断。
17. 证据等级是分层工具，不是降维工具。不得因为客户、份额、ASP、产品代际或排产尚未达到公告事实级别，就把它们从主线中删除或压成风险提示。高重要市场变量必须先按“如果市场口径成立，利润斜率如何变陡”充分展开，再标注证据等级、兑现阶段和验证路径。终稿的默认气质应是“市场交易主线 + 证据护栏”，不是“审计底稿 + 风险罗列”。
18. 质量标准是“参考报告级别”，不是“保守正确”。如果用户要求“像参考报告一样优秀”或表达对风格收敛不满，必须显著提高主笔强度：标题更像结论而非栏目名，正文更像投资人推演而非资料汇编，市场口径必须进入利润桥和上行情景，绿色爆点必须和红色证伪同等重要。终稿允许有进攻性判断和清晰偏向，但每个偏向都要给出兑现路径、变量增强/减弱和验证指标。

## Research Posture

正式报告要同时有进攻性和边界感。先把市场正在交易的上行模型讲完整，再用证据等级和终审 QA 约束它；不要先把变量降噪成低弹性结论。供应链平台股研究的默认写作顺序是：先解释市场为什么愿意重新定价公司，再解释哪些证据已经兑现、哪些处于半兑现、哪些仍是产业链口径。

- 公告事实确定底线：收入、利润、现金流、产能、项目进展、官方披露产品和客户边界。
- 市场口径打开弹性：产品代际、客户份额、订单排产、ASP、毛利率、单机价值量、关键原料、良率、供应商清单。市场口径不等于事实，但它经常是股价和利润斜率正在交易的真正变量，必须进入主线、利润桥、敏感性和跟踪体系。
- 情景假设负责天花板：用可调整的数字或区间展示“如果市场口径成立，利润中枢怎样抬高；如果不成立，回到哪一档”。
- 中期结构 QA 负责防止报告过薄和缺章；终审事实 QA 负责证伪和事实边界。QA 不要在每一段反复弱化主线，也不得把终稿改成“证据不足所以全部降级”的保守综述；它只负责标注兑现阶段、替代解释和验证指标。

### Aggressive Spine Rule

终稿必须先建立“进攻型主线”，再进入证据分层。写作时优先回答：

```text
市场正在重新定价什么？
这家公司为什么不只是普通周期修复？
哪些变量会让利润斜率变陡，而不是只让收入线性增长？
公告事实证明了哪一段，市场口径打开哪一段天花板，下一季看什么来确认？
```

高重要但未公告验证的变量，例如产品代际升级、终端平台迭代、客户份额、订单排产、关键原料锁定、瓶颈产能、单机/单柜/单车/单套价值量、ASP、毛利率或良率，不得只写成“待验证风险”。必须写成“市场押注 -> 利润传导 -> 成立时升级 -> 不成立时降级 -> 验证指标”。如果最终报告读起来像审计底稿、只剩事实边界和风险提示，说明本 skill 执行失败。

### Reference-Grade Report Rule

优秀参考报告的核心不是“把所有风险都说一遍”，而是让读者看完后知道市场为什么愿意交易这家公司、未来利润桥在哪里变陡、什么信号会让逻辑升级。执行本 skill 时，默认按参考报告级别写作：

```text
1. 先给主笔判断：公司发生了什么身份切换，市场错在哪里，利润桥为什么会变。
2. 再给产业链机制：终端平台/技术路线如何传导到价值量、ASP、毛利率、份额和现金流。
3. 再给经营承接：订单、产能、客户认证、原料、良率和子公司如何把预期变成报表。
4. 再给三情景：底线不是“保守废话”，基准不是“中性描述”，上行必须有明确触发器。
5. 最后给反方：反方是用来定义证伪条件，不是用来削弱主线气势。
```

如果某段只是在“事实 A + 风险 B + 仍需验证 C”之间打转，没有回答“如果变量兑现，收入/ASP/毛利率/份额/现金流/归母利润怎么变”，必须重写。参考报告级别的段落应该读起来像一条可交易、可跟踪、可证伪的投资逻辑，而不是“正确但没用”的资料摘要。

### Section Title And Judgment Rule

章节标题必须直接给判断，不能用自我缓冲、两边讨好或泛化安全表达。标题要尽量同时包含“变量 + 机制 + 结论方向”，让读者只看目录就能知道研究者在押什么、利润桥怎么变、跟踪点在哪里。

禁用或强烈避免以下标题句式：

```text
不是 A，而是 B
不只是 A，也可能 B
A 不是小标签
A 不是只有风险
A 不是无敌，但具备条件
A 代表公司下注
A 仍需验证
A 有望受益
```

这些写法通常说明判断过于收敛、变量没有落到利润桥、标题在替正文找退路。应改成更有交易含义的结论型标题：

```text
下一代产品渗透率决定 ASP 斜率，验证点是客户认证和分部毛利率
关键原料紧缺把涨价从成本压力变成高端交付权重
下游制造子公司放量把集团逻辑从材料供应推进到订单验证
近端扩产贡献未来 4-8 季度供给弹性，远期基地决定下一轮高端产能天花板
海外龙头锁定最高端份额是核心反证，公司必须用认证、份额和高端毛利率突破
```

标题不要求预测一定兑现，但必须写清“如果兑现，什么财务变量会变”。如果标题只表达态度、框架或谨慎程度，而没有产品代际、ASP、毛利率、份额、产能、现金流、少数股东或验证指标中的至少一个，应重写。

写作上优先使用可判断表达：

- 如果只是普通产品放量，收入会涨但利润弹性有限；如果高端产品规模量产，同样出货面积会带来更高 ASP 和毛利。
- 如果原材料涨价能顺利转嫁，是利润放大器；如果需求降温或客户压价，就是利润挤压器。
- 如果扩产先于客户导入，会先体现为折旧、费用和爬坡成本；如果客户认证和产能利用率同步提升，才会变成收入和利润上限。
- 如果平台地位只体现在综合份额，它支撑规模和交付；如果进入最高端细分客户清单，它才支撑更高毛利率和更强客户粘性。
- 如果新一代架构/平台要求材料等级跃迁，同一台设备/同一柜/同一车/同一套系统的价值量会被重新计算；若公司进入核心材料清单，天花板上移，若只停留在普通产品供应，逻辑回到周期修复。

表格只能做索引或 QA。凡表格中出现的核心变量，正文必须至少用一段解释其需求来源、公司受益环节、财务传导、变量增强/减弱时的结论变化。不得用表格替代正文推理。

### Market Variable Writing Rule

对所有高重要市场变量，终稿必须采用以下写法：

```text
市场口径说什么：A、B、C。
证据等级是什么：公告事实、券商口径、产业链口径、专家纪要、媒体/自媒体或弱线索。
兑现阶段是什么：主题映射、预期型、半兑现、经营兑现、还是证伪。
为什么市场交易它：它抬高 TAM、单机价值量、ASP、毛利率、客户粘性、份额，还是利润年份切换。
如果成立：收入、ASP、毛利率、份额、客户替换成本、良率、现金流或归母利润会怎样变。
如果不成立：逻辑降级成什么，是普通周期修复、补库涨价、低端放量、扩产先行、成本挤压，还是主线证伪。
如何验证：公司公告、客户认证、订单金额、分产品收入/毛利率、产销量、现金流、应收存货、合同负债、资本开支或客户 capex 中哪一项能验证。
```

正文表达上，证据等级应嵌入变量段落，而不是在段首不断刹车。推荐写法是：“市场正在押 X；若 X 成立，利润桥通过 A/B/C 变陡；目前公告已验证 D，产业链口径指向 E，尚需 F 验证。”不推荐写法是：“X 尚未公告验证，因此只能谨慎看待。”

## Agent Policy

正式研究必须触发多角色研究，但角色权重要向主线集中。6 个纵向主攻 agent 负责推进上行逻辑；3 个横向增强角色负责把报告从摘要拉成正式深度；QA 分两轮执行：中期结构 QA 先防止缺章和过薄，终审事实 QA 再做边界和证伪。

真实 subagent 是首选执行载体。当前环境、宿主规则和 subagent 工具能力都允许时，必须启动 subagent 分工完成以下 6 个纵向主攻角色、3 个横向增强角色和 2 轮 QA：

- 市场变量主线 agent：负责主动搜集券商、产业链、专家纪要、交易热词、竞品、客户 capex 和终端路线，形成市场交易变量扩表。周期信息只回答“为什么新增需求会穿透到公司产品”，不得单独扩写成行业景气综述。
- 架构/技术路线与天花板 agent：负责终端平台/架构路线、产品代际、单机/单柜/单车/单套价值量、利润天花板抬升。AI 方向要优先查 GB200/GB300/Rubin/Feynman、M7/M8/M9/M10、800G/1.6T、HBM/先进封装等。供应链地位只保留能解释 ASP、毛利率、份额、认证门槛和客户替换成本的部分。
- 原材料价格链 agent：负责关键原料成本结构、价格涨跌、供应紧缺、同行调价、成本转嫁、库存和毛利率传导。
- 产能升级与第二曲线 agent：负责基地地图、扩产/投产/爬坡、产品定位、海外基地、控股/参股公司、第二利润曲线和协同关系。承接动作中的扩产、设备、海外基地、原料锁定归入本 agent。
- 订单出货与经营验证 agent：负责订单、排产、出货、收入确认、毛利率、应收、存货、现金流、客户认证、供应链地位和承接动作验证。地位不单独写“龙头/平台”话术，只回答是否能变成订单稳定性、ASP、毛利率、份额、现金流和客户替换成本。
- 利润天花板/利润桥 agent：负责存量利润底盘、新周期新增利润、利润天花板、利润斜率、扩张成本、市场变量的利润传导、并表/权益法/少数股东处理。
- 行业周期与供需格局 agent：负责市场规模、增速、周期位置、价格周期、供需缺口、行业集中度、资本开支和供给释放节奏。必须回答“为什么行业 beta 足够大、当前在周期哪一段、供需缺口能持续多久、价格如何传导到公司”。
- 竞争格局与客户认证链 agent：负责同行对比、技术代际、产能、客户、认证状态、份额、替代风险和公司壁垒。必须建立“终端客户 -> ODM/OEM/系统厂 -> PCB/零部件厂 -> 材料/设备供应商”的客户链，区分事实、券商口径和产业链口径。
- 跟踪体系 agent：负责季度跟踪日历、红色警报信号、关键经营指标、敏感性边界和证伪触发条件。不得只写风险列表，必须给出“哪个指标在什么区间说明逻辑增强/降级”。
- 中期结构 QA：在终稿前、提纲后执行，专门检查是否缺行业、技术、竞争、客户、产能、利润模型、跟踪体系，是否过度摘要化，是否缺产品代际矩阵和可计算利润模型。
- 终审事实 QA：终稿前集中寻找证据缺口、替代解释、市场变量误用、周期见顶风险、把扩产当订单、把收入当利润、把市场口径当事实等问题。它只负责校准和证伪，不作为平权章节抢主线。

每个研究 agent 都必须输出本角色识别到的“市场变量交付块”和“上行情景交付块”。只要涉及产品代际、客户份额、ASP、毛利率、订单/排产、良率、核心客户认证、关键原料、产能利用率、单机价值量或第二曲线，即使证据弱，也要写清：市场口径说什么、证据等级、若成立如何改变收入/毛利率/份额/现金流/利润天花板、若不成立逻辑降级成什么、应在终稿哪个章节保留。

如果当前环境、宿主规则或工具能力不允许真实 subagent，必须用“分阶段文件化”的方式模拟上述 6 个纵向主攻角色、3 个横向增强角色和 2 轮 QA：每个阶段单独产出文件，下一阶段读取文件后再继续，不要只在上下文里滚动压缩。不要把“无法启动真实 subagent”当作跳过角色的理由。

任何角色无法获取足够信息时，仍要输出市场正在交易的假设、缺少的数据、合理的情景边界和验证路径，不能跳过该角色。

### Step 3.5: 市场交易变量扩表

在分发底座 agent 前，先写入 `<标的简称或代码>_market_variables_map.md`。这是正式研究的方向盘，必须回答：

- 市场当前最关心的 5-12 个变量是什么，包括终端架构/平台、产品代际、客户份额、ASP、毛利率、订单排产、关键原料、产能瓶颈、第二曲线和利润天花板。
- 每个变量来自哪里：公告事实、行业/客户一手资料、券商口径、产业链口径、专家纪要、媒体/自媒体、合理推断或情景假设。
- 每个变量为什么抬高天花板或改变利润斜率。
- 若变量成立，收入、ASP、毛利率、份额、客户替换成本、良率、现金流、少数股东权益或投资收益如何变化。
- 若变量不成立，逻辑降级成哪一档。
- 哪个 agent 负责继续深挖，终稿哪个章节必须保留。

若标的处于技术路线快速迭代的供应链环节，必须建立通用链路：下游平台迭代 -> 性能瓶颈变化 -> 零部件/材料规格升级 -> 单机价值量、ASP、毛利率或用量变化 -> 供应商认证、份额、良率、产能约束 -> 订单、收入、利润和现金流兑现。详细方法见 [references/technology-route-value-chain.md](references/technology-route-value-chain.md)。

若存在明确产品代际或材料等级升级，例如 M6/M7 -> M8/M9、400G -> 800G/1.6T、风冷 -> 液冷、普通连接器 -> 高速连接器、普通电源 -> 高功率密度电源，证据台账必须保留价格/价值量口径：

```text
上一代产品价格、价值量或毛利率口径。
新一代产品价格、价值量或毛利率口径。
价差或毛利率差来自哪里：性能、认证、良率、稀缺、客户粘性，还是短期供需。
公司能拿到多少：公司事实、市场口径、情景假设必须分开。
对收入和毛利的压力测试：占比提升 5/10/20 个百分点，或单机价值量变化，对收入/毛利/现金流有什么影响。
扣减项：原材料、良率爬坡、折旧、费用、少数股东损益、投资收益重复计算。
```

证据优先级和数据路径见 [references/data-path.md](references/data-path.md)。

| 市场变量 | 中间文件是否覆盖 | 终稿是否保留 | 若删除，原因 |
| --- | --- | --- | --- |

凡“市场变量”属于高重要口径，例如客户份额、产品代际、ASP、毛利率、订单排产、良率、核心客户认证、上游关键材料锁定等，`终稿是否保留` 原则上必须为“保留”。只有当变量与公司相关性弱、已被反证、或与另一个正文变量合并表达时，才允许删除，并必须写明。

### Step 5.5: 终稿扩写蓝图

在写 `final_report.md` 前，必须先写 `<标的简称或代码>_final_report_expansion_plan.md`。这是防止终稿变成摘要的施工图，不是读者最终交付。扩写蓝图必须逐章列出：

| 终稿章节 | 对应中间文件 | 必写数据/表格 | 必写正文机制 | 变量增强/减弱 | 验证指标 | 目标表达 |
| --- | --- | --- | --- | --- | --- | --- |

硬要求：

- 每个核心章节至少绑定 1 个中间文件，不能凭上下文记忆写。
- 每章先列“要搬进正文的事实/数字/客户链/竞争链/模型”，再列“要写的人话机制”。
- 每章必须写清变量增强和变量减弱，不允许只写“后续关注”。
- 每章目标表达必须明确：用最少但足够的篇幅讲清“变量 -> 财务传导 -> 验证/证伪”。不要用固定字数倒逼扩写；若一章已经讲清楚，不必继续补字。
- 扩写蓝图必须先做“变量复杂度动态规划”，再判定终稿复杂度：`compact`、`standard`、`complex` 或 `long-form`。不要先定字数再塞内容；应先数清高重要市场变量、产品代际/技术路线、客户链/竞争链、利润线和证据缺口，再决定章节和闸门。
- 变量复杂度判定口径：

| 复杂度 | 适用条件 | 终稿目标 | 闸门用法 |
| --- | --- | --- | --- |
| `compact` | 高重要市场变量不超过 3 个，产品代际不超过 2 条，利润线不超过 1 条，且用户要求简洁或证据有限 | 8,000-10,000 中文字，10 个左右二级章节 | `--profile compact` |
| `standard` | 变量数量中等，主线集中，但仍需完整行业、产品、客户、产能、利润和跟踪 | 10,000-14,000 中文字，10-12 个二级章节 | 默认或 `--profile standard` |
| `complex` | 高重要市场变量不少于 6 个，或产品代际不少于 4 条，或客户链不少于 3 条，或利润线不少于 3 条；类似江波龙这类平台型多变量标的默认进入此档 | 12,000-16,000 中文字，12-16 个二级章节 | `--profile complex` 或 `--profile auto --market-variable-count N ...` |
| `long-form` | 用户明确要求长篇深度、券商深度版、对标 25-40 页报告，或参考报告本身要求长篇 | 15,000-25,000 中文字 | `--profile long-form` |

- 若使用 `--profile auto`，扩写蓝图必须记录输入变量数，例如高重要市场变量数、产品代际数、客户链数、利润线数。复杂度不是机械字数，而是“每个变量是否需要一段机制解释、一个验证指标和一个证伪条件”。
- 对 `standard` 和 `complex`，不得把终稿压到闸门最低线附近。若正文刚好接近最低字数但仍有高重要变量只用表格表达，必须补成机制段落；若所有变量已经讲清，则应合并薄章节，而不是为补字重复展开。
- 扩写蓝图未完成，不允许写 `final_report.md`。

### Step 6: 面向普通投资人的终稿

读取全部中间文件后，再写 `<标的简称或代码>_final_report.md`。这是主终点之一，也是最终面向用户的核心交付。

终稿必须写成正式研究报告，而不是研究摘要。默认正式报告应以清楚、完整、可跟踪为目标，通常 10,000-14,000 中文字；只有在变量少、证据有限或用户要求精简时，才使用 8,000-10,000 中文字的 `compact` 模式；当公司变量复杂、产品线和客户链多、利润桥需要多情景解释时，优先使用 12,000-16,000 中文字的 `complex` 模式；只有在用户要求“长篇深度版/券商深度版/对标 25-40 页报告”时，才扩展到 15,000-25,000 中文字。终稿不单列正式估值章节，不放目标价、目标市值、PE/PEG/SOTP 表，也不写买卖建议；但可以写利润中枢、利润年份切换、市场正在交易的预期差和经营情景。正式估值材料统一放在 `valuation_handoff`。终稿第一性目标是讲清“为什么市场正在交易它，以及变量成立时利润天花板如何打开”；事实边界和反方观点服务于这条主线，不能喧宾夺主。

#### Final Report Anti-Summary Gate

正式深度报告模式下，`final_report.md` 生成后必须立即运行反摘要闸门；未通过时不得向用户宣称终稿完成，必须先补写再复跑。默认闸门对应 `standard` 正式报告，要求至少 10,000 个中文字符、10 个二级章节、8 张表、2 个深章节、核心词覆盖，并限制薄章节比例。闸门用于防止摘要化、缺章和“贴最低线交付”，不用于鼓励堆字数。运行闸门时优先使用 profile：

```bash
# 默认 standard
python3 /Users/a/.codex/skills/supply-chain-agentic-research/scripts/final_report_gate.py \
  "/absolute/path/<标的>_final_report.md"

# 变量驱动自动分档
python3 /Users/a/.codex/skills/supply-chain-agentic-research/scripts/final_report_gate.py \
  --profile auto \
  --market-variable-count 10 \
  --product-generation-count 6 \
  --customer-chain-count 4 \
  --profit-line-count 3 \
  "/absolute/path/<标的>_final_report.md"
```

若 `report_outline` 已明确判定为 `compact`，可运行 `--profile compact`；若判定为 `complex`，运行 `--profile complex`；若用户明确要求“长篇深度版/券商深度版/对标 25-40 页报告”，运行 `--profile long-form`。仍允许显式传入 `--min-cjk` 等参数覆盖 profile，但必须在扩写蓝图记录原因。

闸门失败的处理方式：

- 字数不足：先检查是否选错复杂度。复杂标的若只写到 `compact` 长度，通常不是“足够简洁”，而是缺少产品、客户、利润桥或跟踪台阶；应补机制和变量传导。若简单标的内容已经讲清，不要为了字数硬补，应在 `report_outline` 标注 `compact` 并用精简闸门。
- 薄章节过多：逐个检查薄章节是否真的缺内容。若薄是因为逻辑简单且已讲清，可以合并章节；若缺内容，再补“数据表解释 + 机制传导 + 变量增强/减弱 + 验证指标”。
- 概念组缺失：补相应章节，不允许只在摘要或来源处提一次关键词。
- 财务/验证信号不足：补收入、ASP、毛利率、份额、费用、折旧、良率、少数股东、投资收益、现金流或验证指标中的至少两类。
- 失败项必须记录到 `<标的>_final_report_expansion_plan.md` 的“闸门补写记录”中。

补写时必须执行冗余检查：若同一变量已经在前文清楚解释，后文只需要引用结论和新增验证点，不要重复展开同一套“市场押注 -> 财务传导 -> 证伪”。

反摘要闸门之前还要做人工结构检查：

- 每个核心二级章节必须有“结论段 + 数据/证据表 + 机制解释 + 变量增强/减弱 + 验证指标”，不能只有 2-3 段概括。
- 表格不能替代正文。每张核心表格后至少有一段解释“这张表如何改变收入、ASP、毛利率、份额、现金流或归母利润判断”。
- 至少 3 个章节必须显式展开可计算或半可计算模型：产品代际/价值量、订单到现金流、利润桥/敏感性。
- 若章节写完后读起来像“模块结论摘要”，必须回读对应中间文件，把事实、数据、客户链、竞争链、利润传导和证伪路径补进正文。

终稿结构优先使用下列顺序；若内容重叠，可以合并章节，但不能省略对应逻辑：

1. 核心结论与研究框架：先写市场重定价主线，公司是预期型、半兑现型、兑现型供应链平台成长，还是仍停留在主题映射/周期修复；一句话主线、关键变量、利润斜率、证据等级和结论强度。
2. 先讲人话：这家公司在产业链里到底做什么，收入和利润来自哪些环节。
3. 行业空间与周期位置：市场规模、增速、过去周期、当前阶段、供需缺口、价格周期和行业 beta。
4. 市场交易变量与主线：市场为什么重新关注它，正在押哪些订单、技术路线、价格、份额和利润变量。
5. 技术路线与产品代际矩阵：下游平台迭代如何改变性能瓶颈、产品代际、关键材料/零部件规格、单机/单柜/单套价值量和供应商门槛。
6. 竞争格局与客户认证链：同行、产品代际、产能、认证、客户、份额和替代风险；不得只写龙头地位。
7. 公司业务版图与产能地图：主业、子公司、参股公司、海外基地、普通产能、高端产能和瓶颈产能。
8. 原材料价格链与传导：关键原料价格、供需、转嫁机制、库存重估、毛利率留存和现金流压力。
9. 订单出货与经营验证：客户 capex 是否穿透到订单，供应链地位是否变成订单稳定性、ASP、毛利率、份额、现金流和客户替换成本。
10. 第二利润曲线和协同：控股/参股公司、上游材料、下游制造、汽车/先进封装/海外等第二曲线如何贡献利润，是否会被少数股东或权益法稀释。
11. 利润中枢和可计算利润桥：收入拆分、产品占比、ASP/价值量、毛利率、费用、折旧、良率、少数股东、投资收益、所得税、现金流和归母贡献。
12. 情景与敏感性：底线/基准/上行三情景；高端占比、ASP、毛利率、产能利用率、良率、原材料、少数股东和现金流敏感性。
13. 跟踪日历、绿色爆点与红色警报：未来 4-8 个季度要看什么，哪些信号说明增强、等待或证伪。跟踪体系必须先写绿色增强信号，再写黄灯和红灯，避免输出只有风险没有爆点。
14. 终审 QA、反方观点和最终判断：乐观派押什么、悲观派怀疑什么、哪些证据会让结论升级或降级。

上面的终稿结构是内容顺序，不是标题模板。正式写作时不得机械使用“行业空间与周期位置”“技术路线与产品代际矩阵”“竞争格局与客户认证链”这类低判断标题；必须改写为结论型标题。例如：

```text
低判断标题：技术路线：下一代产品不是小标签，而是价值量重新分配
结论型标题：下一代产品渗透率每提升一档，ASP、认证壁垒和毛利率弹性同步放大

低判断标题：原材料涨价不是只有风险，也可能是顺价权测试
结论型标题：关键原料紧缺决定顺价留存率，是毛利率能否维持高位的核心

低判断标题：竞争格局：公司不是无敌，但已经具备挑战高端份额的条件
结论型标题：海外龙头仍卡最高端份额，公司突破信号只能来自客户认证、份额和高端毛利率

低判断标题：多基地扩产不是订单，但代表公司对高端材料周期的下注
结论型标题：近端扩产验证未来 4-8 季度订单承接，远期基地决定下一轮高端产能天花板
```

每个核心章节必须遵循同一写法：

```text
本章核心观点。
数据/证据表。
市场押注 -> 财务传导 -> 已兑现证据 -> 半兑现/未验证口径 -> 反证/替代解释 -> 判断强度。
变量增强时如何影响收入、ASP、毛利率、份额、现金流或归母利润。
变量减弱时逻辑降级成什么。
下一步验证指标。
```

终稿写作必须执行以下 QA：

```text
1. 搜索“后续关注、仍需验证、不能写成事实、证据不足、待验证”等词。
2. 若这些词出现在核心变量段落，必须检查是否先写清“变量成立时如何抬高天花板或利润斜率”。若没有，必须改写成“市场押注 -> 成立时利润传导 -> 未验证部分 -> 验证指标”，不得只保留谨慎提示。
3. 搜索“高端化、国产替代、结构升级、平台型”等抽象词。
4. 若这些词没有伴随价格、价值量、毛利率、良率、份额、现金流或客户替换成本解释，必须重写。
5. 搜索证据台账和提纲中的高重要市场变量名称，例如产品代际、客户名/代称、ASP、毛利率、份额、订单、排产、良率、关键原料等。若这些词没有进入 final_report 正文，必须补写或在市场变量覆盖 QA 中解释合并/删除原因。
6. 搜索“估值、目标价、目标市值、PE、PEG、SOTP、隐含利润、赔率、定价”等词。
7. 若出现在 final_report，必须删除或移至 valuation_handoff。
8. 搜索“行业、竞争、客户、认证、产品代际、利润桥、敏感性、跟踪日历、红色警报”。若缺任一类正文内容，必须补写，不能只放在中间文件。
9. 检查每个核心章节是否至少包含一个数据表或模型表、一个解释性段落、一个变量增强/减弱判断和一个验证指标。
10. 搜索“市场正在交易、利润斜率、天花板、绿色信号、上行情景、半兑现、兑现”。若终稿缺少这些进攻型表达，或只剩事实边界和风险提示，必须重写核心结论和市场变量章节。
11. 单独检查标题。搜索“不是、并非、不只是、不仅、但、具备、有望、下注、测试、仍需验证、分析、概览、梳理”等词。若这些词出现在一级或二级标题中，通常说明标题在缓冲判断，必须改成“变量 + 财务传导 + 结论方向”的标题。
12. 运行 `scripts/final_report_gate.py`。若失败，必须把失败项写入 `<标的>_final_report_expansion_plan.md` 的“闸门补写记录”，补写后复跑。
13. 正式导出 PDF/HTML 前，必须运行 `$research-report-publication-editor` 的 publication hygiene gate，清除导出痕迹、skill/subagent/工具名、终稿自述和提示词式内部审稿语言；若存在 HIGH 问题，不得声称正式版完成。最终回复用户时说明两个闸门是否通过。
```

## Output Discipline

最终回复用户时，除非用户要求只看终稿，否则要说明：

- 新生成了哪些中间文件。
- 哪些 agent / 阶段完成了哪些任务。
- 6 个纵向主攻模块、3 个横向增强模块、2 轮 QA 和 1 条技术路线横向链路如何被覆盖，最好给出覆盖表。
- 是否完成了行业-技术-竞争-客户-产能-利润模型-跟踪体系全链条成稿。
- `final_report` 是否已经完成。
- 终稿文件路径。
