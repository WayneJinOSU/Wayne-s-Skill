# Agentic 底盘型成长股研究

## Overview

这是一个独立主控 skill，直接编排底盘型成长股研究的私有 modules，并通过多角色研究、中间文件、进攻型市场重定价主线、行业经济性与蛋糕分配、竞争/客户验证链、中期进攻 QA、利润桥、跟踪体系、终审事实 QA 和投资人写作控制输出质量。`final_report` 是本 skill 的主终点；它先把市场为什么重新定价、上行变量如何传导到利润斜率讲完整，再由证据边界和反方审查校准收敛。终稿负责把旧业务底盘、行业为什么扩容、公司为什么能分到蛋糕、第二曲线把上限抬到哪里、平台复用为什么让跨界可信、竞争客户验证如何支撑份额、承接动作如何变成收入利润、利润中枢和证伪路径讲清楚。正式估值由 `$growth-stock-valuation` 接手；本 skill 只准备估值接力输入，`final_report` 不写目标价、目标市值、PE/PEG/SOTP、买卖评级或明确投资建议。

核心判断是：

```text
旧业务底盘够硬 + 市场重定价变量清楚 + 行业经济性改善带来渗透率提升 + TAM/SAM/SOM 不混用 + 公司份额提升路径清楚 + 第二曲线抬高天花板 + 平台能力可复用 + 竞争客户验证不缺位 + 承接动作进入经营兑现 + 利润桥走通 + 证据边界和证伪路径明确
```

本 skill 的目标不是生成高密度摘要，也不是把模块框架机械填满，而是通过多角色研究和中间产物，产出普通投资人能读懂、投资人能跟踪、反方能检验的正式研究报告：市场为什么重新定价，公司旧业务给了什么下限，行业为什么扩容，行业增长的钱为什么不是所有人平分，公司凭什么拿更多，第二曲线把上限抬到哪里，平台复用为什么让跨界可信，客户和竞争证据如何验证新身份，承接动作如何变成收入和利润，利润桥在哪些变量上变陡，哪些信号会让主线升级或证伪。

## When to use

适合用户要求：

- 深度分析一只 A 股制造业、AI 算力、数据中心、新能源、机器人、先进制造、半导体设备/材料等底盘型成长公司。
- 判断公司是否发生身份切换、行业经济性拐点、第二成长曲线成型、份额提升、平台能力复用、竞争客户验证、订单/客户/产能承接、利润中枢上移，并把证据、推理、市场口径和证伪点讲清楚。
- 不只要核心结论，还要把旧业务底盘、行业经济性与蛋糕分配、新业务天花板、平台复用、承接动作、利润桥和后续跟踪讲清楚。
- 需要用 subagent / 多角色 / 多阶段研究避免一次性摘要化输出。

不适合：

- 银行、地产、保险、纯资源周期、早期创新药、纯消费品牌。
- 无收入利润底盘、只有概念没有客户/订单/产能/研发承接的题材股。
- 用户只要“一句话看法”。如果用户要求“快速判断”“先粗筛”，可以使用轻量模式，但必须说明未完成完整 agentic 流程。

## Non-negotiables

正式研究必须同时满足以下要求：

1. 完整覆盖 7 个核心私有研究模块、3 个横向护栏模块、1 个中期进攻 QA、1 个终审事实 QA 和 1 个估值接力输入：
   - 市场重定价主线
   - 旧业务底盘
   - 第二成长曲线
   - 行业经济性与蛋糕分配
   - 平台复用能力
   - 承接动作
   - 利润桥
   - 横向护栏：竞争/客户验证链
   - 横向护栏：跟踪体系
   - 横向护栏：证据边界与验证路径
   - 中期进攻 QA：检查是否缺行业 beta、竞争客户链、市场变量、上行情景、利润斜率、跟踪体系、证据边界和估值接力输入
   - 终审事实 QA：反方审查、事实边界、替代解释和证伪检查，只负责校准，不负责削弱主线气势
   - 估值接力输入：只写正式估值需要的事实底盘、市场变量、利润情景和证伪点，不写估值结论
   研究模块由本 skill 私有管理，模块文件放在 `modules/` 目录下；中期进攻 QA 不单独做 module，写入 `<标的>_midterm_structure_review.md`。本 skill 不再调用或依赖外部 common 模块。正式估值阶段调用 `$growth-stock-valuation`。
2. 不允许直接输出终稿。必须先形成中间研究产物。
3. 每个核心判断必须写清楚：

```text
证据 -> 推理 -> 反证/替代解释 -> 判断强度
```

4. 必须区分：
   - 公告或财报事实
   - 结构化行情和财务数据
   - 客户、订单、产能或行业一手资料
   - 市场口径
   - 合理推断
   - 情景假设
5. 最终报告必须面向一般投资人，解释术语、铺设推理台阶，不能只堆结论、概念和表格。
6. 对市场正在交易但公告尚未充分验证的变量，不允许简单删除，也不允许只放在证据台账、脚注或反方审查里。必须降级保留为市场口径、券商假设、产业链线索或待验证变量，并写清证据等级、利润影响和验证路径。
7. 不要在正文主线阶段过早压低变量。先把市场正在交易什么、行业为什么扩容、公司为什么能分到蛋糕、利润如何变厚写清楚；最后再统一校准证据边界和验证路径。证据分层服务于判断，不替代判断。
8. 终稿必须先建立进攻型重定价主线：市场到底在押什么，公司为什么不只是旧业务修复或普通题材映射，哪些变量会让利润斜率变陡，公告事实验证了哪一段，下一季看什么来确认。不得把高重要市场变量先写成“证据不足、后续关注”后就结束。
9. 第二成长曲线必须回答“抬收入、抬毛利率、抬客户粘性、抬平台身份，还是只抬题材热度”。不得把行业空间直接等同于公司利润。
10. 行业经济性与蛋糕分配必须回答：行业扩容来自成本下降、效率提升、性能突破、认证完成、客户批量导入、供给瓶颈解除还是政策/资本开支；TAM 只说明想象空间，SAM 说明可服务方向，SOM 才允许进入公司收入和利润桥。
11. 平台复用必须拆技术、客户、工艺、产能、供应链、认证体系、组织能力和经济性曲线。不能只写“协同”“平台化”“复用能力强”。
12. 承接动作必须区分扩产、融资、设备、客户验证、订单、量产和财报兑现。扩产不是订单，订单不是利润，收入增长不是利润中枢永久上移。
13. 竞争/客户验证链必须回答：竞争对手是否也能拿到同一新业务机会、客户验证处于送样/认证/小批/量产/份额提升哪一段、公司优势来自技术、成本、交付、客户关系、认证周期还是平台复用。不得只写“客户优质、竞争格局好”。
14. 利润桥默认做轻量利润中枢和利润斜率，不替代正式估值。必须拆清旧业务底盘利润、行业扩容中公司可获取份额、第二曲线新增利润、平台复用收益、扩张费用/折旧/财务成本/爬坡损耗、少数股东权益和现金流质量。
15. 中期进攻 QA 和终审事实 QA 不得把报告重新拉回保守摘要。中期 QA 首先检查“上行变量、利润斜率、客户份额、毛利率弹性、第二曲线天花板和跟踪触发器是否写透”；终审 QA 再检查事实边界、替代解释和证伪条件。
16. 搜索层必须高召回。优先组合公告网站、交易所、公司官网、投资者关系、客户/竞品公告、券商研报摘要、行业机构、结构化数据、`$wencai-query`、`$tushare` 和普通 web search；进入证据收集前必须检查 `GEMINI_API_KEY`。若可用，必须运行 [scripts/gemini_google_search.py](scripts/gemini_google_search.py) 调用 Google Search 作为线索发现器；若不可用，必须在证据台账记录 `Gemini未运行原因`。Gemini/Google 搜索结果只能进入证据台账的“市场口径/线索”层，必须回到原始公告、研报 PDF、公司/客户/竞品资料或结构化数据复核后，才能支撑正文判断。
17. `final_report` 不放正式估值章节，不写目标价、目标市值、PE/PEG/SOTP、买卖建议或“当前贵不贵”的估值结论；这些统一进入 `valuation_handoff` 或 `$growth-stock-valuation`。

## Research Posture

正式报告要同时有进攻性和边界感。先把市场正在重新定价的变量讲完整，再用证据等级和反方审查约束它；不要先把变量降噪成低弹性结论。

- 公告事实确定底线：收入、利润、现金流、客户、产能、研发和管理层披露。
- 市场口径打开弹性：第二曲线空间、客户导入、产品放量、份额、ASP/毛利率、平台身份、核心资源、组织投入。市场口径不等于事实，但经常是股价和利润斜率正在交易的真正变量，必须进入主线、竞争/客户验证链、利润桥、敏感性和跟踪体系。
- 情景假设负责天花板：用底线/基准/上行三层情景说明“如果市场口径成立，利润中枢怎样抬高；如果不成立，回到哪一档”。
- 中期进攻 QA 负责防止报告缺章、过薄和低进攻性：在提纲前检查市场变量、竞争客户链、承接动作、上行情景、利润斜率和跟踪体系是否都进入正文逻辑。它不是风控闸门，不得把高重要市场口径从正文删回证据台账。
- 反方审查负责证伪和事实边界，不负责把终稿改成“证据不足所以全部降级”的保守综述。反方输出必须同时保留乐观派押注、悲观派怀疑、验证信号和降级路径。

### Aggressive Repricing Spine Rule

终稿必须先建立“进攻型重定价主线”，再进入证据分层。写作时优先回答：

```text
市场正在重新定价什么？
这家公司为什么不只是旧业务修复、普通扩产或概念映射？
哪些变量会让利润斜率变陡，而不是只让收入线性增长？
公告事实证明了哪一段，市场口径打开哪一段天花板，下一季看什么来确认？
```

高重要但未公告验证的变量，例如第二曲线客户导入、产品放量、份额提升、ASP/毛利率改善、平台复用、核心资源、产能爬坡、团队投入或少数股东权益变化，不得只写成“待验证风险”。必须写成“市场押注 -> 利润传导 -> 成立时升级 -> 不成立时降级 -> 验证指标”。如果终稿读起来像审计底稿、只剩事实边界和风险提示，说明本 skill 执行失败。

### Section Title And Judgment Rule

章节标题必须尽量给出判断，不能只写栏目名或自我缓冲。标题优先包含“变量 + 机制 + 结论方向”，让读者只看目录就能知道研究者在押什么、利润桥怎么变、跟踪点在哪里。

禁用或强烈避免以下标题句式：

```text
A 仍需验证
A 有望受益
A 不是小标签
A 具备想象空间
A 值得后续关注
```

应改成更有交易含义的结论型标题：

```text
第二曲线若进入客户小批量，收入弹性通过产品占比和费用摊薄放大
平台复用的关键不是概念协同，而是认证、良率和交付周期能否迁移
行业蛋糕扩大只有进入 SOM 才能进利润桥，验证点是份额、订单和分业务毛利率
扩产先于订单会先消耗现金流，客户认证同步推进才会变成利润斜率
```

### Evidence Grading Timing Rule

证据边界校准放在研究末端统一执行。数据采集和主线写作阶段可以积极吸收公告、券商、行业报告、公司交流、客户/竞品线索和市场口径；终稿前再把它们分成事实底盘、经营验证、市场假设和弱线索。分级的目的不是删除变量，也不是把终稿改成审计底稿，而是决定变量在正文中的身份：事实结论、经营验证、利润弹性假设、跟踪变量或反方风险。

### Market Variable Writing Rule

对所有高重要市场变量，终稿和模块中必须采用以下写法：

```text
变量：
市场口径说什么：
证据等级：
兑现阶段：主题映射、预期型、半兑现、经营兑现，还是证伪。
如果成立：收入、毛利率、份额、客户粘性、费用率、现金流或归母利润会怎样变。
如果不成立：逻辑降级成什么，是旧业务修复、普通扩产、概念包装、成本挤压，还是主线证伪。
验证指标：公告、客户验证、订单、分业务收入/毛利率、产销量、现金流、应收存货、资本开支或管理层指引中的哪一项能验证。
终稿保留方式：
```

不合格写法：

```text
市场口径不能写成公司事实，因此不纳入正文。
新业务收入、客户和毛利率未披露，所以只能后续关注。
券商数据证据等级低，放在证据台账即可。
```

合格写法：

```text
市场正在押第二曲线客户导入和高毛利产品放量；若成立，利润桥会通过收入增量、毛利率提升和费用摊薄变陡。当前公告只验证了产能/研发/客户合作的一部分，市场口径仍需用订单、分业务毛利率和现金流验证；若长期没有经营兑现，逻辑降级为旧业务修复加概念映射。
```

## Agent Policy

正式研究必须触发多角色研究。市场重定价主线、旧业务底盘、第二曲线、行业经济性与蛋糕分配、平台复用、竞争/客户验证链、承接动作/经营验证、利润桥、跟踪体系、证据边界、终审事实 QA/反方审查这些研究/审查角色，以及中期进攻 QA，是质量控制机制，不是可选项。

真实 subagent 是首选执行载体。当前环境、宿主规则和 subagent 工具能力都允许时，必须启动 subagent 分工完成这些角色：

- 市场重定价主线 agent：负责市场正在交易什么、身份切换在哪里、预期差来自哪里、哪些变量决定利润斜率。
- 旧业务底盘 agent：负责收入利润、现金流、客户基础、行业地位、财务质量和下限支撑。
- 第二曲线 agent：负责新业务定义、产业链位置、价值量、产品结构、客户粘性和天花板变化；不得把行业 TAM 直接写成公司利润。
- 行业经济性与蛋糕分配 agent：负责行业为什么扩容、渗透率为何提升、TAM/SAM/SOM 如何分层、公司市占率如何变化、哪些收入和利润可以合理进入公司利润桥。
- 平台复用 agent：负责技术、客户、工艺、产能、供应链、认证体系、组织能力和经济性曲线如何复用，以及跨界可信度。
- 竞争/客户验证链 agent：负责同行对比、客户验证阶段、份额、替代风险、认证周期、客户粘性和公司优势来源。先回答“客户验证/份额提升若成立，收入、毛利率和客户粘性如何上修”，再回答替代风险，防止只写行业空间、平台复用或风险清单。
- 承接动作/经营验证 agent：负责扩产、融资、设备、客户验证、订单、量产、研发投入、海外基地、人员变化和现金流验证。
- 利润桥 agent：负责旧业务底盘利润、行业扩容中公司可获取份额、第二曲线新增利润、平台复用收益、扩张成本、市场变量的利润传导、并表/权益法/少数股东处理。
- 跟踪体系 agent：负责把主线、行业经济性、份额变化和利润桥转成 2-4 个季度的红黄绿灯、关键验证指标、升级/降级条件和证伪触发。
- 证据边界 agent：负责在终稿前把关键变量分成 A/B/C/D 证据等级，说明事实、经营验证、市场假设和弱线索分别如何保留；输出应嵌入核心变量、利润桥和跟踪体系，不强制单独写成终稿大章，不能把重要市场变量简单删除。
- 中期进攻 QA：在写终稿提纲前执行，检查是否缺竞争客户链、行业 beta、行业经济性、TAM/SAM/SOM、公司份额、市场变量正文保留、上行情景、利润斜率、跟踪体系、证据边界和估值接力输入；若缺失，必须先补中间文件再写 `final_report`。QA 的第一目标是把主线写厚，不是把变量删薄。
- 终审事实 QA/反方审查 agent：终稿前集中寻找证据缺口、替代解释、市场变量误用、把扩产当订单、把收入当利润、普通投资人理解断点和证伪条件。它只负责校准和证伪，不作为平权章节抢主线，不得把高重要市场变量简单删回风险提示。

每个研究 agent 都必须输出本角色识别到的“市场变量交付块”和“上行情景交付块”。只要涉及新业务收入、客户份额、订单/量产、产品放量、毛利率、平台复用、研发团队、产能爬坡、核心资源或少数股东权益，即使证据弱，也要写清：市场口径说什么、证据等级、若成立如何改变收入/毛利率/费用率/现金流/利润中枢、若不成立逻辑降级成什么、应在终稿哪个章节保留。

上行情景交付块必须包含：

```text
上行变量：
利润传导：
触发条件：
降级条件：
验证指标：
终稿保留章节：
```

如果当前环境、宿主规则或工具能力不允许真实 subagent，必须用“分阶段文件化”的方式模拟这些独立研究角色：每个阶段单独产出文件，下一阶段读取文件后再继续，不要只在上下文里滚动压缩。不要把“无法启动真实 subagent”当作跳过角色的理由。

任何角色无法获取足够信息时，仍要输出市场正在交易的假设、缺少的数据、合理的情景边界和验证路径，不能跳过该角色。

### Step 2: 主线假设与适用性分诊

先写入 `<标的简称或代码>_question.md`：

- 市场正在交易什么：身份重估、第二曲线、行业经济性改善、渗透率提升、公司份额变化、利润中枢上移、订单兑现、旧业务错杀修复或平台能力复用。
- 需要验证的 3 个核心问题。
- 先写“进攻型主线假设”：如果市场变量成立，利润中枢如何上移；如果不成立，逻辑降级到哪一档。
- 公司是否适用底盘型成长股框架；如不适用，停止完整流程并说明更合适框架。

### Step 3: 证据收集

写入 `<标的简称或代码>_evidence_index.md`，按可获得性覆盖：

- 公告原文：年报、季报、业绩预告/快报、募投、定增/可转债、重大合同、关联交易、问询回复、业绩说明会。
- 结构化数据：`$tushare` 拉行情、市值、财务三表、财务指标、现金流、历史行情；市值和估值类字段只用于市场阶段判断和 `valuation_handoff`，不在终稿形成正式估值结论。
- 市场位置：`$wencai-query` 查概念、同行排名、题材口径、阶段涨幅、市场关注点。
- 公司官网、交易所互动、投资者关系、客户/竞品公告、行业资料和可靠新闻。
- 市场增量情报：券商研报核心假设、产业链调研、专家问答、交易热词、新业务客户和竞品进展。此类资料不能直接等同事实，但必须用于识别市场真正交易的边际变量。
- 行业空间与份额口径：券商/行业报告里的 TAM/SAM/SOM、渗透率、价格/成本/效率曲线、供需缺口、竞争份额和利润率假设。此类数据必须标注来源和口径，但不能因不是公告事实就从主线删除。
- Gemini Google Search 条件强制增强：先检查 `GEMINI_API_KEY`。若可用，使用 `scripts/gemini_google_search.py` 分别搜索市场重定价变量、行业经济性/TAM/SAM/SOM、客户/竞品验证、平台复用、承接动作和利润桥线索，至少覆盖 3 组与标的最相关的查询；若不可用或脚本失败，必须在本台账记录 `Gemini未运行原因`、错误信息和替代搜索路径。输出只作为线索索引，必须记录查询词、模型、日期、返回来源和待复核问题；不得把模型总结直接当事实。

证据台账必须识别“市场变量”，但表格只能作为索引或 QA，不能替代正文推理。对每个高重要市场变量，必须写清：变量、市场口径、证据等级、成立时利润传导、不成立时降级路径、验证指标和终稿保留方式。

市场重定价主线写入 `<标的简称或代码>_market_repricing_spine.md`：

- 市场正在重新定价什么：旧业务错杀、第二曲线、利润中枢上移、平台身份切换、订单兑现还是情绪映射。
- 市场真正交易的 5-10 个变量是什么，哪些已被公告验证，哪些只是市场/券商/产业链口径。
- 每个变量如何改变收入、毛利率、费用率、客户粘性、现金流或归母利润。
- 变量不成立时，逻辑降级到哪一档。

旧业务底盘研究写入 `<标的简称或代码>_base_business_floor.md`：

- 主营和收入结构，旧业务收入、利润和现金流是否稳定。
- 毛利率、费用率、净利率、应收、存货和经营现金流。
- 客户基础、行业地位、竞争壁垒和下限支撑。
- 若第二曲线不兑现，旧业务能提供什么底线，哪些因素会削弱底线。

第二成长曲线研究写入 `<标的简称或代码>_second_curve_ceiling.md`：

- 新业务定义、周期阶段、产业链位置和天花板性质；行业空间只做初步识别，TAM/SAM/SOM 细拆交给 `<标的简称或代码>_industry_space_share.md`。
- 新业务是抬收入、抬利润率、抬客户粘性、抬平台身份，还是只抬题材热度。
- 单机价值量、产品结构、客户份额、毛利率和天花板变化。
- 若市场口径成立/不成立，利润中枢如何上修或降级。

行业经济性与蛋糕分配写入 `<标的简称或代码>_industry_space_share.md`：

- 行业为什么扩容：成本下降、效率提升、性能突破、认证完成、客户批量导入、供给瓶颈解除、政策/资本开支或其他变量。
- TAM/SAM/SOM 分层：TAM 只说明想象空间，SAM 说明 3-5 年可服务方向，SOM 说明公司在份额、客户、产能、订单和竞争约束下可能拿到的收入和利润。
- 市场占有率和份额来源：当前份额、目标份额、可提升原因、竞争对手反应、价格压力和客户多供风险。
- 贡献利润与天花板：哪些空间只能做长期想象，哪些能进入 2-4 个季度跟踪，哪些可以进入利润桥，并说明利润率、费用和现金流约束。

平台复用研究写入 `<标的简称或代码>_platform_reuse.md`：

- 旧业务能力：技术、客户、工艺、产能、供应链、认证体系、组织能力和经济性曲线。
- 新业务所需能力：哪些能复用，哪些需要重新建设。
- 复用链条如何降低兑现难度，如何转化为订单稳定性、费用率、毛利率、成本下降、良率爬坡、客户粘性或渗透率提升。
- 是否存在硬跨界、能力断层或概念包装。

竞争/客户验证链写入 `<标的简称或代码>_competition_customer_validation.md`：

- 竞争格局：主要同行、技术路线、产品代际、客户结构、成本、交付、毛利率和扩产节奏。
- 客户验证链：接触/送样/认证/小批/量产/份额提升分别有哪些证据，哪些只是市场口径。
- 替代风险：竞争对手通过低价、交期、客户双供、多供、技术路线替代时，公司逻辑如何降级。
- 竞争客户变量如何影响收入、毛利率、费用率、客户粘性、订单稳定性和利润斜率。

承接动作研究写入 `<标的简称或代码>_execution_signals.md`：

- 扩产、融资、设备、客户验证、订单、量产、海外基地、研发投入和人员变化是否已经发生。
- 区分资本开支、客户验证、订单、量产和财报兑现，不得把扩产直接当订单。
- 承接动作如何影响收入、毛利率、费用、折旧、现金流和良率。
- 若承接动作没有进入订单/收入/利润，逻辑如何降级。

利润桥研究写入 `<标的简称或代码>_profit_bridge.md`：

- 旧业务底盘利润、行业扩容中公司可获取份额、第二曲线新增利润、平台复用收益、扩张成本的方向性拆解。
- 默认输出轻量利润中枢：方向性区间、关键变量和敏感性；完整正式估值交给 `$growth-stock-valuation`。
- 保守、中性、乐观三种情景可以简化为“利润斜率情景”：分别由哪些市场变量触发，并说明证据强度。
- 必须处理并表利润、权益法收益、少数股东损益、非经常性损益和现金流质量，避免重复计算。

跟踪体系写入 `<标的简称或代码>_tracking_dashboard.md`：

- 后续 2-4 个季度看什么：订单、收入、毛利率、现金流、应收存货、扩产进度、客户导入、产品放量、价格/成本/效率曲线、渗透率、份额和证据等级升级。
- 红黄绿灯：哪些指标说明逻辑增强，哪些说明只是旧业务修复或概念映射，哪些说明主线证伪。
- 对利润桥最敏感的 3-5 个变量和跟踪频率。

证据边界与验证路径写入 `<标的简称或代码>_evidence_grading.md`，在终稿前、反方审查前后均可执行，但不得前置成删变量机制，也不要求终稿单独成章：

- A 级：公告、财报、交易所问询/回复、监管文件、结构化财务和行情数据。可作为事实底盘。
- B 级：公司 IR、官网、业绩说明会、客户/竞品公告、招投标、认证、产品发布、公开订单或量产线索。可作为经营验证。
- C 级：券商深度报告、行业报告、份额估计、TAM/SAM/SOM、公开专家观点。作为市场假设、情景变量和利润弹性来源。
- D 级：未交叉验证的产业链调研、专家访谈、未披露订单线索、交易热词、弱口径。作为跟踪触发器和待验证变量；若被公告、客户/竞品公开信息、券商/行业报告或多源口径交叉验证，可升为 C/B。
- 对每个核心变量说明：当前等级、为什么不能写成事实、为什么仍应保留、成立时利润传导、不成立时降级路径和验证指标。

三表建模接力输入写入 `<标的简称或代码>_financial_model_handoff.md`，用于先调用 `$financial-modeling` 建三表/FCF，再接力 `$dcf-valuation` 或 `$growth-stock-valuation` 校验；它不是估值结论，不进入 `final_report`：

- 从旧业务底盘、第二曲线、行业空间份额、执行信号、利润桥、跟踪体系和证据分级中抽取可建模驱动，分清公司事实、市场口径和自有假设。
- 至少覆盖收入拆分/增速、毛利率、费用率、折旧摊销、capex/revenue、DSO/DIO/DPO、税率、利息/债务、少数股东/投资收益、经营现金流和 FCF 质量。
- 必须输出驱动项表：`驱动项 | 业务来源 | 历史锚/来源 | 保守 | 中性 | 乐观 | 证据等级 | 财报验证指标 | 降级路径`。
- 必须输出 DCF-ready UFCF 桥所需字段：`EBIT*(1-Tax)+D&A-Capex-ΔNWC`；缺数据写“待 financial-modeling 补数/待财报补充”，不得硬编精确数字。
- 若某变量只能支撑利润情景、不能支撑三表预测，必须标为“情景压力测试”，不得写成基准模型假设。

估值接力输入写入 `<标的简称或代码>_valuation_handoff.md`：

- 当前事实底盘：旧业务利润、现金流、第二曲线进展、行业经济性、公司份额、承接动作、利润桥情景。
- 市场正在交易的核心变量和证据等级。
- 保守/中性/乐观利润情景的触发条件和关键证伪指标。
- PEG-ready 利润锚：2025A/2026E/2027E/2028E 的一致预期、自有中性、自有乐观、YoY 增速、CAGR 区间、利润口径和证据等级；缺失项明确写“待估值阶段补充”。
- 估值倍数接力判断：把完整投研报告中的旧业务下限、市场重定价主线、第二曲线可信度、行业经济性、份额提升、平台复用、利润质量、现金流和证伪压力，翻译成基础 PEG 档位、乐观 PEG 档位、是否允许 1.5x 以上、是否允许切换 2028E 及必要触发条件。
- PEG 修正项：低基数、非经常性、并表/少数股东、现金流质量、订单前置、毛利率弹性、旧业务周期性和第二曲线兑现度如何影响 PEG 折价或溢价。
- 正式估值需要回答的 3-5 个问题。不得写目标价、目标市值、买卖建议或最终估值结论。

| 年份 | 利润口径 | 一致预期 | 自有中性 | 自有乐观 | YoY 增速 | CAGR 区间 | 证据等级 | 备注 |
| --- | --- | ---: | ---: | ---: | ---: | --- | --- | --- |
| 2025A | 归母/扣非/经营利润 |  |  |  |  |  | A |  |
| 2026E | 归母/扣非/经营利润 |  |  |  |  |  | A/B/C |  |
| 2027E | 归母/扣非/经营利润 |  |  |  |  |  | A/B/C |  |

建议 PEG 档位只作为估值 skill 的倍数锚输入：周期修复且利润质量一般通常 `0.8-1.0x`；收入兑现但利润待确认通常 `1.0-1.2x`；半兑现成长且客户/订单链较强通常 `1.1-1.3x`；兑现型平台龙头且现金流和毛利率同步通常 `1.2-1.4x`；产业主升浪稀缺龙头通常 `1.3-1.5x`；`>1.5x` 只能标为高潮期或趋势上沿，必须绑定强触发条件。

### Step 5.5: 终稿扩写蓝图

在写 `final_report.md` 前，必须先写 `<标的简称或代码>_final_report_expansion_plan.md`。这是防止终稿变成摘要的施工图，不是读者最终交付。扩写蓝图必须逐章列出：

| 终稿章节 | 对应中间文件 | 必写中间素材 | 必写数据/表格 | 必写正文机制 | 变量增强/减弱 | 验证指标 | 目标表达 |
| --- | --- | --- | --- | --- | --- | --- | --- |

硬要求：

- 每个核心章节至少绑定 1 个中间文件，不能凭上下文记忆写。
- 每个核心章节必须列出 3-8 条要吸收的中间素材，素材至少覆盖事实底盘、市场变量、行业经济性、平台复用、竞争客户链、承接动作、利润传导、反证/降级路径、跟踪指标中的三类。若某章只能列出 1-2 条素材，说明该章不具备正式成稿条件，必须回到对应中间文件补写或并入相邻章节。
- 每章先列“要搬进正文的事实/数字/客户链/竞争链/利润桥素材”，再列“要写的人话机制”。
- 每章必须写清变量增强和变量减弱，不允许只写“后续关注”。
- 每章目标表达必须明确：用最少但足够的篇幅讲清“变量 -> 财务传导 -> 验证/证伪”。不要用固定字数倒逼扩写；若一章已经讲清楚，不必继续补字。
- 扩写蓝图必须先做变量复杂度判定，再决定终稿复杂度和反摘要闸门 profile。

| 复杂度 | 适用条件 | 终稿目标 | 闸门用法 |
| --- | --- | --- | --- |
| `compact` | 高重要市场变量不超过 3 个，第二曲线不超过 1 条，客户/竞争链不超过 1 条，利润线不超过 1 条，且用户要求简洁或证据有限 | 6,000-8,000 中文字，8 个左右二级章节 | `--profile compact` |
| `standard` | 变量数量中等，主线集中，但仍需完整旧业务、第二曲线、行业蛋糕、平台复用、竞争客户、利润桥和跟踪 | 8,000-12,000 中文字，8-10 个二级章节 | 默认或 `--profile standard` |
| `complex` | 高重要市场变量不少于 5 个，或第二曲线不少于 2 条，或客户/竞争链不少于 3 条，或利润线不少于 3 条 | 12,000-16,000 中文字，10-12 个二级章节 | `--profile complex` 或 `--profile auto --market-variable-count N ...` |
| `long-form` | 用户明确要求长篇深度、券商深度版、对标 25-40 页报告，或参考报告本身要求长篇 | 15,000-22,000 中文字 | `--profile long-form` |

- 若使用 `--profile auto`，扩写蓝图必须记录输入变量数：高重要市场变量数、第二曲线数、客户/竞争链数、利润线数。
- 对 `standard` 和 `complex`，不得把终稿压到闸门最低线附近。若正文刚好接近最低字数但仍有高重要变量只用表格表达，必须补成机制段落；若所有变量已经讲清，则应合并薄章节，而不是为补字重复展开。
- 扩写蓝图未完成，不允许写 `final_report.md`。

### Step 6: 面向普通投资人的终稿

读取全部中间文件后，再写 `<标的简称或代码>_final_report.md`。

`final_report` 是主笔重构，不是模块摘要。终稿必须按 `<标的简称或代码>_final_report_expansion_plan.md` 写成正式研究报告，而不是研究备忘录；核心章节必须使用 Markdown H2 标题 `##`，以便反摘要闸门识别。默认正式报告以 8,000-12,000 中文字的 `standard` 模式为基准；变量少、证据有限或用户明确要求精简时，才使用 6,000-8,000 中文字的 `compact` 模式；变量复杂、第二曲线/客户链/利润线较多时，优先使用 12,000-16,000 中文字的 `complex` 模式；只有用户要求“长篇深度版/券商深度版/对标 25-40 页报告”时，才扩展到 15,000-22,000 中文字。终稿不单列正式估值章节，不放目标价、目标市值、PE/PEG/SOTP 表，也不写买卖建议；正式估值材料统一放在 `valuation_handoff`。

终稿写作必须执行“防摘要化主笔规则”：

1. 不得把 `final_report` 写成“模块结论汇总”。每个核心章节必须重新组织成投资人能读懂的论证链：事实是什么，市场为什么交易它，为什么能影响收入/毛利率/费用率/现金流/利润中枢，反证是什么，下一步看什么。
2. 表格只能作为段落后的收束、索引或对比，不能替代正文推理。若删掉某个表格后章节无法读懂，说明正文过薄，必须补段落。
3. 每个核心章节至少包含以下五类内容中的三类：公告或财报事实、市场口径或产业链变量、利润传导、反证/替代解释、验证指标。市场重定价、第二曲线、行业经济性、竞争客户链、承接动作和利润桥章节必须同时包含利润传导和反证/降级路径。
4. 第二成长曲线章节必须逐方向回答：该方向是抬收入、抬毛利率、抬客户粘性、抬平台身份，还是只抬题材热度；不能只写“空间大、前景好、仍需关注”。
5. 行业经济性章节必须逐应用场景回答：行业为什么扩容、谁分蛋糕、公司凭什么分到、哪些能进入 SOM、哪些只能停留在 TAM/SAM。
6. 竞争/客户验证链章节必须逐客户或应用说明验证阶段：接触、送样、认证、小批、量产或份额提升；不能只写“客户优质、竞争格局好”。
7. 利润桥章节必须写成桥，不是结论表：从收入增量到毛利率、费用率、折旧/财务费用/爬坡损耗、少数股东、现金流质量逐层传导。
8. 证据边界不能把终稿压成保守摘要。对高重要市场变量，即使不能写成事实，也必须保留“若成立怎样升级、若不成立怎样降级、用什么验证”的段落。
9. 章节标题必须有判断密度。若标题只表达“有望、关注、仍需验证、空间较大”，必须改成“变量如何改变收入/毛利率/费用率/客户粘性/现金流/利润中枢，以及验证点是什么”。

终稿结构优先使用下列顺序；若内容重叠，可以合并章节，但不能省略对应逻辑。证据边界与验证路径应嵌入核心变量、利润桥、跟踪体系或反方观点，不强制单独成章：

1. 一句话结论：公司是预期型、半兑现型、兑现型，还是仍停留在主题映射/旧业务修复。
2. 先讲人话：这家公司到底靠什么赚钱。
3. 市场为什么重新定价：第二曲线、平台身份、利润中枢或旧业务错杀。
4. 旧业务底盘：下限在哪里，现金流和客户基础是否可靠。
5. 第二成长曲线：上限来自哪里，是否抬收入、抬毛利率或抬平台身份。
6. 行业经济性与蛋糕分配：渗透率为什么提升，TAM/SAM/SOM 如何分层，公司份额、利润贡献和未来天花板如何约束。
7. 平台复用：为什么不是硬跨界，旧能力和经济性曲线如何迁移到新业务。
8. 竞争/客户验证链：同行能不能复制，公司客户验证走到哪一步，份额和替代风险如何影响利润斜率。
9. 承接动作：扩产、客户验证、订单、量产和研发投入是否匹配。
10. 利润中枢：利润斜率、关键敏感变量和证伪条件。
11. 跟踪体系：未来几个季度看什么，红黄绿灯如何判断。
12. 反方观点、证据边界、证伪点和最终判断。

终稿写作必须执行以下 QA：

```text
1. 搜索“后续关注、仍需验证、不能写成事实、证据不足、待验证”等词。
2. 若这些词出现在核心变量段落，必须改写成“变量增强/减弱时结论如何变化”。
3. 搜索“平台化、协同、国产替代、第二曲线、生态化”等抽象词。
4. 若这些词没有伴随收入、毛利率、费用率、客户、订单、现金流或复用链条解释，必须重写。
5. 搜索“估值、目标价、目标市值、PE、PEG、SOTP、隐含利润、赔率、定价”等词。
6. 若出现在 final_report 中形成估值结论，必须删除或移至 valuation_handoff。
7. 随机抽查 3 个核心章节：如果章节主要由表格和短结论组成，必须补写段落。
8. 搜索一级标题：若标题没有变量、机制或结论方向，必须重写为结论型标题。
9. 对照 final_report_expansion_plan 的章节素材映射表：若中间文件中的关键证据只在终稿中被一句话概括，必须恢复为完整论证。
10. 运行 `scripts/final_report_gate.py`。若失败，必须把失败项写入 `<标的>_final_report_expansion_plan.md` 的“闸门补写记录”，补写后复跑。最终回复用户时说明闸门是否通过。
```

#### Final Report Anti-Summary Gate

正式深度报告模式下，`final_report.md` 生成后必须立即运行反摘要闸门；未通过时不得向用户宣称终稿完成，必须先补写再复跑：

```bash
python3 /Users/a/.codex/skills/chassis-growth-agentic-research/scripts/final_report_gate.py \
  "/absolute/path/<标的>_final_report.md"
```

若 `report_outline` 已明确判定为 `compact`，运行 `--profile compact`；若判定为 `complex`，运行 `--profile complex`；若用户明确要求“长篇深度版/券商深度版/对标 25-40 页报告”，运行 `--profile long-form`。变量驱动自动分档示例：

```bash
python3 /Users/a/.codex/skills/chassis-growth-agentic-research/scripts/final_report_gate.py \
  --profile auto \
  --market-variable-count 6 \
  --second-curve-count 2 \
  --customer-chain-count 3 \
  --profit-line-count 3 \
  "/absolute/path/<标的>_final_report.md"
```

闸门失败时，优先补缺失的机制、证据、利润传导、验证指标或证伪条件；不得为了通过闸门添加重复段落。失败项必须记录到 `<标的>_final_report_expansion_plan.md` 的“闸门补写记录”中。

正式导出 PDF/HTML 前，必须运行 `$research-report-publication-editor` 的 publication hygiene gate，清除导出痕迹、skill/subagent/工具名、终稿自述和提示词式内部审稿语言；若存在 HIGH 问题，不得声称正式版完成。
