# Agentic 供应链平台型成长股研究

## Overview

这是一个独立主控 skill，直接编排供应链平台型成长股研究的六个核心模块，并通过多角色研究、中间文件、反方审查和投资人写作控制输出质量。

核心判断仍然是：

```text
产业周期够大 + 供应链位置关键 + 订单出货领先 + 利润平台换台阶 + 当前估值未过度透支
```

本 skill 的目标不是生成核心摘要，而是通过多角色研究、中间文件、反方审查和投资人写作，把供应链平台股的逻辑讲完整：下游周期为什么成立、公司为什么不是普通代工、订单如何变收入、收入如何变利润、当前估值到底提前定价了什么。

## When to use

适合用户要求：

- 深度分析一只处在 AI 算力、数据中心、新能源、机器人、先进封装、半导体设备/材料等大产业周期中的供应链平台型公司。
- 判断公司是否发生身份切换、订单出货放量、利润中枢上移和远期估值重估。
- 不只要核心结论，还要把产业周期、供应链地位、订单出货、利润桥和估值逻辑讲清楚。
- 需要用 subagent / 多角色 / 多阶段研究避免一次性摘要化输出。

不适合：

- 银行、地产、保险、纯消费品牌、纯资源周期、纯软件互联网、早期创新药。
- 无收入利润底盘、只有概念没有订单和产能承接的题材股。
- 用户只要“一句话看法”。如果用户要求“快速判断”“先粗筛”，可以使用轻量模式，但必须说明未完成完整 agentic 流程。

## Non-negotiables

正式研究必须同时满足以下要求：

1. 完整覆盖供应链平台研究的 6 个核心模块：
   - 产业周期
   - 供应链地位
   - 订单出货
   - 承接动作
   - 利润桥
   - 远期估值
   对应子 skill 分别是 `$supply-chain-cycle-capex`、`$supply-chain-position-moat`、`$supply-chain-orders-shipments`、`$common-growth-execution-signals`、`$common-growth-profit-bridge`、`$common-growth-forward-valuation`。能通过 skill 机制调用时，应优先调用；不能直接调用时，必须复用这些子 skill 的问题清单并完整覆盖同名问题。
2. 不允许直接输出终稿。必须先形成中间研究产物。
3. 每个强判断必须写清楚：

```text
证据 -> 推理 -> 反证/替代解释 -> 判断强度
```

4. 必须区分：
   - 公告或财报事实
   - 客户 capex / 行业数据
   - 订单、排产、出货或海关口径
   - 结构化行情和财务数据
   - 市场口径
   - 合理推断
   - 情景假设
5. 最终报告必须面向一般投资人，解释术语、铺设推理台阶，不能只堆概念、数据和结论。

## Agent Policy

正式研究必须触发多角色研究。产业周期、供应链地位、订单出货、利润中枢与估值、反方审查这 5 个研究角色是质量控制机制，不是可选项。

真实 subagent 是首选执行载体。当前环境、宿主规则和 subagent 工具能力都允许时，必须启动 subagent 分工完成这 5 个角色：

- 产业周期与客户 Capex agent：负责下游需求、客户资本开支、产品平台迭代、周期阶段和见顶风险。
- 供应链地位与壁垒 agent：负责产业链位置、客户认证、交付壁垒、可替代性、议价权和客户集中风险。
- 订单出货与经营质量 agent：负责订单、排产、出货、收入确认、毛利率、应收、存货、现金流和承接动作。
- 利润中枢与估值 agent：负责利润桥、旧平台利润、新周期新增利润、扩张成本、远期 PE 和估值透支。
- 反方审查 agent：专门寻找证据缺口、替代解释、估值误导、周期见顶风险和普通投资人理解断点。

如果当前环境、宿主规则或工具能力不允许真实 subagent，必须用“分阶段文件化”的方式模拟 5 个独立研究角色：每个阶段单独产出文件，下一阶段读取文件后再继续，不要只在上下文里滚动压缩。不要把“无法启动真实 subagent”当作跳过角色的理由。

任何角色无法获取足够信息时，必须输出“证据不足”、缺少的数据、因此需要降级的结论，不能跳过该角色。

## Workflow

### Step 1: 建立研究目录

在当前任务目录下建立中间产物目录：

```text
research_artifacts/<股票代码或公司简称>/
  <标的简称或代码>_question.md
  <标的简称或代码>_evidence_index.md
  <标的简称或代码>_cycle_capex.md
  <标的简称或代码>_position_moat.md
  <标的简称或代码>_orders_shipments_quality.md
  <标的简称或代码>_profit_valuation.md
  <标的简称或代码>_skeptic_review.md
  <标的简称或代码>_report_outline.md
  <标的简称或代码>_final_report.md
```

### Step 2: 主线假设与适用性分诊

先写入 `<标的简称或代码>_question.md`：

- 市场正在交易什么：身份切换、订单放量、利润中枢上移、估值透支、错杀修复。
- 需要验证的 3 个核心问题。
- 公司是否适用供应链平台型框架；如不适用，停止完整流程并说明更合适框架。

### Step 3: 证据收集

写入 `<标的简称或代码>_evidence_index.md`，按可获得性覆盖：

- 公告原文：年报、季报、业绩预告/快报、订单、合同、募投、定增、可转债、问询回复、业绩说明会。
- 客户和行业资料：客户财报、资本开支指引、电话会、产品平台路线、行业出货、权威机构或协会数据。
- `$tushare`：股价、市值、PE/PB、财务三表、财务指标、现金流、历史行情。
- `$wencai-query`：题材概念、市场排名、同行筛选、阶段涨幅、事件性条件。
- 海关、出货、排产和产业链跟踪数据；若不是官方来源，必须说明可靠性和口径限制。

证据优先级和数据路径见 [references/data-path.md](references/data-path.md)。

### Step 4: 多角色研究

正式研究中，主控必须覆盖 6 个供应链平台核心模块，但可以由不同 agent 或阶段分工完成。

主控必须在 `<标的简称或代码>_report_outline.md` 或终稿前保留一张覆盖表：

| 核心模块 | 对应子 skill | 负责角色/阶段 | 结论强度 | 证据缺口 |
| --- | --- | --- | --- | --- |
| 产业周期 | `$supply-chain-cycle-capex` | 周期 |  |  |
| 供应链地位 | `$supply-chain-position-moat` | 地位 |  |  |
| 订单出货 | `$supply-chain-orders-shipments` | 订单/经营质量 |  |  |
| 承接动作 | `$common-growth-execution-signals` | 订单/经营质量 |  |  |
| 利润桥 | `$common-growth-profit-bridge` | 利润/估值 |  |  |
| 远期估值 | `$common-growth-forward-valuation` | 利润/估值/反方 |  |  |

产业周期研究写入 `<标的简称或代码>_cycle_capex.md`：

- 下游需求是否真实爆发，是否仍有 1 到 3 年可见度。
- 客户 capex 是否扩张，是否和公司产品相关。
- 产品平台迭代是否提升价值量、复杂度、ASP 或良率要求。
- 周期处于早期放量、中期扩散、后期拥挤，还是见顶回落。

供应链地位研究写入 `<标的简称或代码>_position_moat.md`：

- 公司是核心平台、关键供应商、重要配套，还是普通供应商。
- 壁垒来自认证、规模、交付、良率、成本、全球产能还是供应链组织。
- 可替代性、客户集中风险和议价权。
- 为什么不能把“大客户供应商”直接等同于“平台型公司”。

订单出货与经营质量研究写入 `<标的简称或代码>_orders_shipments_quality.md`：

- 订单、排产、出货、量产和收入确认证据，并按强/中/弱信号分级。
- 建立订单 -> 排产 -> 出货 -> 收入确认 -> 利润释放 -> 现金回收传导链。
- 检查毛利率、净利率、应收、存货、合同负债、经营现金流。
- 承接动作是否只是扩产，还是已经对应订单、量产和收入利润。

利润中枢与估值研究写入 `<标的简称或代码>_profit_valuation.md`：

- 旧平台利润、新周期新增利润、扩张费用/折旧/财务成本/爬坡损耗。
- 利润中枢是否真的换台阶，还是季度高利润的年化误读。
- 当前市值反推 2025、2026E、2027E、2028E PE。
- 当前估值隐含哪一年、哪一种利润情景，是否已经透支。

反方审查写入 `<标的简称或代码>_skeptic_review.md`：

- 哪些结论证据不足。
- 哪些地方把行业景气直接跳成了公司壁垒。
- 哪些地方把扩产当订单、把订单当利润、把收入增长当利润平台上移。
- 哪些估值口径可能误导普通投资人。
- 最关键 3 到 5 个证伪点。

### Step 5: 写作提纲

写入 `<标的简称或代码>_report_outline.md`，不要直接写终稿。提纲必须回答：

- 读者第一眼需要先理解什么。
- 哪些供应链术语必须用人话解释。
- 哪些结论需要先铺产业周期，再讲公司地位。
- 哪些表格应该保留，哪些应该转成解释性段落。
- 最终报告的 6 到 10 个一级章节。

### Step 6: 面向普通投资人的终稿

读取全部中间文件后，再写 `<标的简称或代码>_final_report.md`。

终稿结构优先使用下列顺序；若内容重叠，可以合并章节，但不能省略对应逻辑：

1. 一句话结论：公司是预期型、半兑现型、兑现型供应链平台成长，还是已经充分定价。
2. 先讲人话：这家公司在产业链里到底做什么。
3. 为什么市场重新关注它：产业周期和身份切换。
4. 下游周期：客户 capex 和产品平台迭代是否支撑未来订单。
5. 供应链地位：为什么不是普通代工或普通配套。
6. 订单出货：订单如何变成收入、利润和现金流。
7. 承接动作与经营质量：扩产、量产、应收、存货、现金流是否匹配。
8. 利润中枢：利润平台是否真正换台阶。
9. 估值：当前市值反映到哪一年利润，是否透支。
10. 反方观点、证伪点和最终判断。

## Output Discipline

最终回复用户时，除非用户要求只看终稿，否则要说明：

- 新生成了哪些中间文件。
- 哪些 agent / 阶段完成了哪些任务。
- 六个供应链平台核心模块如何被覆盖，最好给出覆盖表。
- 终稿文件路径。
