# Final Report Construction

本文件承接 `SKILL.md` 的终稿提纲、扩写蓝图、防摘要化、复杂度分档和反摘要闸门要求。`references/report-writing.md` 负责投资人写作风格；本文件负责执行型硬约束。

## Report Outline

写入 `<标的简称或代码>_report_outline.md`，不要直接写终稿。提纲必须回答：

- 是否已经通过中期进攻 QA；若未通过，必须先补写缺失模块。
- 读者第一眼需要先理解什么。
- `investment_logic_card` 的“投资人首页判断”如何扩写为开篇 800-1200 字，以及 3-6 个主线句如何进入终稿前 30%：核心交易逻辑、市场误定价、最大预期差/核心分歧、稀缺资源或关键瓶颈、第一驱动参数、最小利润模型、利润中枢区间、验证点。
- 全文唯一第一主线是什么；旧业务、第二曲线、行业蛋糕、平台复用、客户验证、承接动作和反方审查分别如何服务、校准或证伪这条主线。
- 底盘型成长股叙事链如何贯穿全文：旧业务给下限，新变量改变利润斜率，公司用平台/产能/客户放大变量，扩产/认证/产品结构决定兑现节奏，折旧、财务费用和现金流决定利润穿透率，竞争和证据边界决定上限与降级路径。
- 哪些概念必须用人话解释。
- 市场真正交易的 3-8 个变量是什么，终稿如何保留。
- 上行情景里的 3-5 个关键触发器是什么，如何进入章节标题、利润桥和跟踪体系。
- 行业扩容、公司份额和利润贡献分别讲到什么层级：TAM 做想象，SAM 做可服务方向，SOM 才进收入利润。
- 竞争/客户验证链中的关键变量如何进入正文，哪些只作为反方或跟踪指标。
- 哪些结论需要先铺业务逻辑，再给财务数字。
- 哪些表格应该保留，哪些应该转成解释性段落。
- 反方审查放在哪些后段章节校准主线，且不得改写前段进攻判断。
- 最终报告的 6-8 个一级章节；`complex` 可放宽到 8-10 个，`long-form` 才允许 10-12 个。模块覆盖不等于终稿章节，第二曲线、行业蛋糕、平台复用、竞争客户链、承接动作和证据边界应优先合并到投资主线中，但对应逻辑不可省略。
- 一级标题必须优先写成结论型标题，至少包含变量、机制或结论方向之一；不能只写“第二曲线”“平台复用”“仍需验证”这类栏目名。
- 终稿复杂度初判：`compact`、`standard`、`complex` 或 `long-form`，并记录高重要市场变量数、第二曲线数、客户/竞争链数和利润线数。

## Expansion Plan

在写 `final_report.md` 前，必须先写轻量 `<标的简称或代码>_final_report_expansion_plan.md`。它不是新增研究报告，而是防止终稿摘要化和漏读证据的施工卡；目标是少写、写准、能直接指导主笔。

扩写蓝图只保留三张短表：

1. 前段进攻主线表：引用 `<prefix>_investment_logic_card.md`，说明“投资人首页判断”如何扩写为开篇 800-1200 字，并列出核心交易逻辑、最大预期差、第一驱动参数、最小利润模型、利润中枢区间、验证点和降级条件。
2. 章节素材表：每个核心章节绑定 3-5 条必须吸收的素材，写明对应中间文件、优先读取的下游摘要块或片段、影响的利润参数、变量增强/减弱和验证指标。
3. 表达转换表：列出本轮会用到的内部编号或过程词，以及它们在 `final_report` 中对应的自然来源表述，例如“公司公告/财报披露”“产业链线索显示”“券商假设的核心分歧在于”。

硬要求：

- 不做大型十列表，不重复搬运模块摘要，不为每章机械填满 8 条素材。
- 每章必须服务第一主线：增强、校准、证伪或提供经营下限；不能映射到利润参数的内容并入相邻章节、移入跟踪项或删除。
- 若某章少于 3 条有效素材，先补写中间文件或合并章节，不要硬写薄章。
- 读取计划只列必要片段：相关 Fact-ID/Lead-ID 行、模块下游摘要块、指定小节或关键词；只有遇到口径冲突、证据缺口或 gate 失败时，才回读全文或原始来源。
- 扩写蓝图必须记录复杂度 profile，供 `final_report_gate.py` 选择参数。
- 扩写蓝图未完成，不允许写 `final_report.md`。

## Complexity Profiles

| 复杂度 | 适用条件 | 终稿目标 | 闸门用法 |
| --- | --- | --- | --- |
| `compact` | 高重要市场变量不超过 3 个，第二曲线不超过 1 条，客户/竞争链不超过 1 条，利润线不超过 1 条，且用户要求简洁或证据有限 | 5,000-7,000 中文字，6-7 个二级章节 | `--profile compact` |
| `standard` | 变量数量中等，主线集中，但仍需完整旧业务、第二曲线、行业蛋糕、平台复用、竞争客户、利润桥和跟踪 | 7,000-10,000 中文字，7-8 个二级章节 | 默认或 `--profile standard` |
| `complex` | 高重要市场变量不少于 5 个，或第二曲线不少于 2 条，或客户/竞争链不少于 3 条，或利润线不少于 3 条 | 10,000-14,000 中文字，8-10 个二级章节 | `--profile complex` 或 `--profile auto --market-variable-count N ...` |
| `long-form` | 用户明确要求长篇深度、券商深度版、对标 25-40 页报告，或参考报告本身要求长篇 | 14,000-20,000 中文字，10-12 个二级章节 | `--profile long-form` |

若使用 `--profile auto`，扩写蓝图必须记录输入变量数：高重要市场变量数、第二曲线数、客户/竞争链数、利润线数。对 `standard` 和 `complex`，不得把终稿压到闸门最低线附近。若正文刚好接近最低字数但仍有高重要变量只用表格表达，必须补成机制段落；若所有变量已经讲清，则应合并薄章节，而不是为补字重复展开。

## Brokerage-Style Final Report Discipline

`final_report` 本身就是研报化交付，不另写 `brokerage_report`。券商风格只改变表达纪律，不改变研究边界：

- 先给结论和核心分歧，再解释证据链；不要先铺大段行业背景。
- 开篇必须扩写 `investment_logic_card` 的“投资人首页判断”，用“我们认为/核心分歧在于/利润弹性来自/后续验证看/若变量弱化则降级到”的判断句组织。
- 用“我们认为/市场分歧在于/关键变量是/该变量当前更适合按某种情景处理/后续验证看什么”组织判断，但不写买卖评级、目标价、目标市值或正式估值结论。
- 章节标题可以像券商研报，也可以保留结论型标题；标题不是框架。框架由“第一驱动参数 -> 利润传导 -> 兑现节奏 -> 证据边界”决定。
- 研究模块是素材池，不是目录。旧业务、行业景气、公司优势、财务质量、催化剂和风险都要围绕第一驱动参数展开。
- 对底盘型成长股，默认采用“旧业务给下限 -> 新变量改变利润斜率 -> 平台/产能/客户放大变量 -> 扩产/认证/产品结构决定兑现节奏 -> 折旧/财务费用/现金流决定穿透率 -> 竞争和证据边界决定上限与降级路径”的顺序。
- 不得默认套用供应链平台报告的“下游需求 -> 技术路线 -> 价值量 -> 客户认证 -> 订单放量 -> 利润兑现”顺序，除非公司确实主要由订单、客户认证和价值量驱动。
- 材料类或玻纤类底盘公司应优先讲“旧主业给下限 -> 新品类单位经济改变利润斜率 -> 可售量或产能利用率放大弹性 -> 折旧/财务费用/现金流决定穿透率 -> 高端认证和同业供给决定乐观情景”。行业需求、客户验证和产能扩张只能服务这条利润斜率主线。
- 正文不得出现 `Fact-ID`、`Lead-ID`、`skill`、`subagent`、`gate`、`handoff`、`final_report`、`brokerage_report`、`扩写蓝图`、`闸门`、`终稿`、`研究阶段候选`、`正确写法`、`普通投资人最容易误判`、`本报告只做` 等内部痕迹。

## Final Report Rules

不要一次性读取全部中间文件。先读取 `investment_logic_card`、`report_outline`、`final_report_expansion_plan`、`midterm_structure_review` 和 `skeptic_review`；`facts_core` 默认只读取扩写蓝图指定的相关 Fact-ID 行。再按扩写蓝图逐章读取对应中间文件的下游摘要块和必要片段。事实定位可以依赖 Fact-ID/Lead-ID，正式正文必须改写为自然来源和研报表达；不要在读者可见文本中暴露内部编号，也不要为了补字重复搬运同一事实。

终稿阶段的默认读取顺序是：

```text
final_report_expansion_plan 读取计划
-> 相关 Fact-ID/Lead-ID 行
-> 模块下游摘要块
-> 指定小节或关键词片段
-> 仅在冲突/缺口/闸门失败时回读全文或原始来源
```

`final_report` 是主笔重构，不是模块摘要，也不是另一个“券商版”副本。终稿必须按 `<标的简称或代码>_final_report_expansion_plan.md` 写成正式研究报告，而不是研究备忘录；开篇必须先扩写 `investment_logic_card` 的“投资人首页判断”，再吸收其他主线字段和事实材料。核心章节必须使用 Markdown H2 标题 `##`，以便反摘要闸门识别。默认正式报告以 7,000-10,000 中文字的 `standard` 模式为基准；变量少、证据有限或用户明确要求精简时，才使用 5,000-7,000 中文字的 `compact` 模式；变量复杂、第二曲线/客户链/利润线较多时，优先使用 10,000-14,000 中文字的 `complex` 模式；只有用户要求“长篇深度版/券商深度版/对标 25-40 页报告”时，才扩展到 14,000-20,000 中文字。终稿不单列正式估值章节，不放目标价、目标市值、PE/PEG/SOTP 表，也不写买卖建议；估值因子、情景准入、模型消费规则、三表/FCF/UFCF 驱动和 DCF-ready 候选字段只作为研究 readiness 或两轮 gate 后 focused post-report handoff，不进入终稿正文估值结论。

终稿写作必须执行“防摘要化主笔规则”：

1. 不得把 `final_report` 写成“模块结论汇总”。每个核心章节必须重新组织成投资人能读懂的论证链：事实是什么，市场为什么交易它，为什么能影响收入/毛利率/费用率/现金流/利润中枢，反证是什么，下一步看什么。
1a. 开篇必须是进攻型主线，不是审计式免责声明。前 30% 篇幅必须至少出现并展开：核心交易逻辑、市场误定价、最大预期差/核心分歧、第一驱动参数、最小利润模型或单位经济、未来 2-3 年利润中枢区间、最关键验证点。可以标注“中/低置信”，但不能用“待验证/后续关注/证据不足”替代主线。
1b. 开篇 800-1200 字必须能脱离全文单独作为投资人首页阅读：包含“我们认为”或等价判断句、第一驱动参数、事实锚、利润传导、验证点和降级条件；不得出现“这轮研究/研究抓手/报告写法/章节/判断不应只问”等研究方法口吻。
1c. 反方审查后置。`skeptic_review` 的内容应进入后段“反方观点、证据边界、证伪点、跟踪体系”中，用来校准置信度和降级路径；不得把终稿第一章标题或第一段改成“有机会但待验证”。
1d. 全文只能有一个第一主线。旧业务、第二曲线、行业蛋糕、平台复用、竞争客户链、承接动作和反方审查必须分别说明自己如何增强、校准或证伪该主线；不得平铺成多个彼此松散的“看点”。
1e. 禁止说教式正文。不得用长段解释行业常识、研究方法、概念定义或“为什么要关注”；每个段落必须落到公司特定事实、核心利润参数、验证指标或证伪条件。术语解释最多一句，且必须服务当前公司和当前利润模型。
1f. 禁止内部过程语言。`final_report` 中不得出现 Fact-ID/Lead-ID、模块名、subagent、gate、handoff、扩写蓝图、闸门、终稿、自我审稿词或“本报告只做/正确写法/普通投资人最容易误判”等提示词残留。
1g. 券商式语言必须清楚但不装腔。优先使用“我们认为”“核心分歧在于”“利润弹性来自”“后续验证看”“若变量弱化则降级到”这类判断句；不得用模板化标题掩盖因果链不完整。
2. 表格只能作为段落后的收束、索引或对比，不能替代正文推理。若删掉某个表格后章节无法读懂，说明正文过薄，必须补段落。
3. 每个核心章节至少包含以下五类内容中的三类：公告或财报事实、市场口径或产业链变量、利润传导、反证/替代解释、验证指标。市场重定价、第二曲线、行业经济性、竞争客户链、承接动作和利润桥章节必须同时包含利润传导和反证/降级路径。
4. 第二成长曲线章节必须逐方向回答：该方向是抬收入、抬毛利率、抬客户粘性、抬平台身份，还是只抬题材热度；不能只写“空间大、前景好、仍需关注”。
5. 行业经济性章节必须逐应用场景回答：行业为什么扩容、谁分蛋糕、公司凭什么分到、哪些能进入 SOM、哪些只能停留在 TAM/SAM。
6. 竞争/客户验证链章节必须逐客户或应用说明验证阶段：接触、送样、认证、小批、量产或份额提升；不能只写“客户优质、竞争格局好”。
7. 利润桥章节必须写成桥，不是结论表：从收入增量到毛利率、费用率、折旧/财务费用/爬坡损耗、少数股东、现金流质量逐层传导。
7a. 利润桥必须保留最小可计算模型：单位经济、数量乘数、兑现节奏、利润中枢区间、敏感性和置信度。公司未披露分项收入/毛利率时，用区间和敏感性替代空白，不得只写“尚未验证”。
8. 证据边界不能把终稿压成保守摘要。对高重要市场变量，即使不能写成事实，也必须保留“若成立怎样升级、若不成立怎样降级、用什么验证”的段落。
9. 章节标题必须有判断密度。若标题只表达“有望、关注、仍需验证、空间较大”，必须改成“变量如何改变收入/毛利率/费用率/客户粘性/现金流/利润中枢，以及验证点是什么”。

终稿结构优先使用下列 7 章，但章节名可以按标的和第一驱动参数重写。若用户给出“投资要点、核心逻辑、业务底盘、行业景气、公司优势、财务分析、催化剂、风险提示、附录”等券商栏目，只能作为表达外壳；真正顺序仍由底盘型叙事链决定。若内容重叠，继续合并；若确实复杂，可把第 4 或第 5 章拆成两章。证据边界与验证路径应嵌入核心变量、利润桥、跟踪体系或反方观点，不强制单独成章：

1. 进攻型结论与核心交易逻辑：合并“一句话结论”和“市场误定价”。公司到底被市场误解成什么，最大预期差/核心分歧在哪里，真正稀缺资源或关键瓶颈是什么，第一驱动参数是什么，若主线成立利润中枢会怎样跃迁；同时标注置信度，但不以风险开头。
2. 最小利润模型与利润中枢：合并“最小利润模型”和“利润中枢校准”。写清单位经济、数量乘数、兑现节奏、2-3 年利润中枢区间、敏感性、现金流质量和归母口径，并排序第一/第二/第三驱动参数。
3. 公司到底靠什么赚钱：合并“先讲人话”和“旧业务底盘”。旧业务给什么下限，新业务从哪里切入，哪些利润来自主业，哪些来自投资收益、权益法或非经常性项目。
4. 行业瓶颈、蛋糕分配与第二曲线：合并“行业经济性”和“第二成长曲线”。行业为什么扩容，TAM/SAM/SOM 如何分层，公司凭什么分到 SOM，新业务是抬收入、抬毛利率、抬客户粘性、抬平台身份，还是只抬题材热度。
5. 公司凭什么拿到：合并“平台复用”“竞争/客户验证链”和“承接动作”。说明旧能力如何迁移，客户验证走到哪一步，同行能不能复制，扩产/订单/量产/研发投入是否匹配利润释放。
6. 关键验证与跟踪体系：未来 2-4 个季度看什么，红黄绿灯如何判断，哪些信号让主线升级、降级或证伪。
7. 反方观点、证据边界与最终判断：只校准置信度、参数下修和降级路径，不推翻前段主线表达。

以下内容原则上不单独成章，除非 `complex` 或 `long-form` 且确有多条独立主线：市场为什么重新定价、旧业务底盘、第二成长曲线、平台复用、承接动作、证据边界、利润中枢校准。它们应进入上述 7 章的对应段落。

## Anti-Summary Gate

正式深度报告模式下，`final_report.md` 生成后必须立即运行反摘要闸门；未通过时不得向用户宣称终稿完成，必须先补写再复跑：

```bash
python3 /Users/a/.codex/skills/chassis-growth-agentic-research/scripts/final_report_gate.py \
  "/absolute/path/<标的>_final_report.md"
```

若 `report_outline` 已明确判定为 `compact`，运行 `--profile compact`；若判定为 `complex`，运行 `--profile complex`；若用户明确要求“长篇深度版/券商深度版/对标 25-40 页报告”，运行 `--profile long-form`。变量驱动自动分档示例：

```bash
python3 /Users/a/.codex/skills/chassis-growth-agentic-research/scripts/final_report_gate.py \
  --profile auto \
  --market-variable-count 6 \
  --second-curve-count 2 \
  --customer-chain-count 3 \
  --profit-line-count 3 \
  "/absolute/path/<标的>_final_report.md"
```

闸门失败时，优先补缺失的机制、证据、利润传导、验证指标或证伪条件；不得为了通过闸门添加重复段落。失败项必须记录到 `<标的>_final_report_expansion_plan.md` 的“闸门补写记录”中。

`final_report_gate.py` 默认只检查前 1500 个字符内是否出现投资人首页判断、利润传导、验证和降级语言；如需调整，可使用 `--front-window`，但不得把首页判断后移来规避开篇要求。

反摘要闸门 PASS 后必须运行 publication hygiene gate，且中优先级问题也需要修正：

```bash
python3 /Users/a/.codex/skills/research-report-publication-editor/scripts/publication_hygiene_gate.py \
  --fail-on medium \
  "/absolute/path/<标的>_final_report.md"
```

若 publication hygiene gate 失败，优先把研究方法口吻改为研报判断句，而不是只删除词语。
