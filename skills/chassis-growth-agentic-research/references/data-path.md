# 数据路径与证据优先级

## 目标

本 skill 不替代 `$tushare`、`$wencai-query`、公告 PDF 处理、外部搜索或 Gemini Google Search 能力。它负责确定证据链、调用顺序、市场变量扩表和口径解释。

正式 PEG 估值交给 `$growth-stock-valuation`。PEG-ready 输入由 `$growth-stock-valuation` 自己确认或生成。

正式 DCF 估值交给 `$dcf-valuation-workflow` 主控 `$financial-modeling` 和 `dcf-model`。DCF-ready 数据包由 `$dcf-valuation-workflow` 调用 `$financial-modeling` 从 post-report `dcf_financial_model_handoff` 拆出。

本文件只说明哪些数据应进入研究阶段候选字段，以及终稿 gate 通过后的 `dcf_financial_model_handoff` 和 `peg_valuation_handoff`。

## 证据优先级

1. 官方公告原文：年报、季报、业绩预告/快报、订单、募投、定增、可转债、重组、关联交易、问询回复、业绩说明会。
2. 结构化数据：`$tushare` 的股价、市值、财务三表、财务指标、现金流、股本、分红和历史行情。市值和估值类字段只用于市场阶段判断、研究阶段候选字段、post-report `peg_valuation_handoff` 或后续估值模型输入，不在终稿形成正式估值结论。
3. `$wencai-query`：题材概念、市场排名、阶段涨幅、同行筛选、事件性条件和快速交叉验证。
4. 公司官网、投资者关系、交易所互动、客户/竞品公告、可靠新闻和行业资料：用于补充业务进展、客户、订单、产品和产业趋势。
5. 市场增量情报：券商研报核心假设、产业链调研、专家问答、交易热词、新业务客户线索、同行/竞品进展和客户认证线索。此类资料不能写成公告事实，但必须用于识别市场真正交易的边际变量；多源交叉验证的产业链线索可以从弱线索升级为市场假设或经营验证。
6. 行业空间与份额口径：券商/行业报告里的 TAM/SAM/SOM、渗透率、价格/成本/效率曲线、供需缺口、竞争份额和利润率假设。此类资料必须标明来源和口径，作为市场假设、情景变量或跟踪变量进入主线。
7. Gemini Google Search 线索：只在常规来源无法覆盖关键变量时使用。若 `GEMINI_API_KEY` 可用，可用 [../scripts/gemini_google_search.py](../scripts/gemini_google_search.py) 做 1-2 组缺口型来源发现；公告财报、投关、三表、行情、常规新闻和已知研报列表不使用 Gemini 重复搜索。Gemini 输出只能作为“待复核线索”，必须回到原始公告、研报 PDF、公司/客户/竞品公开材料或结构化数据复核。

公告原文与结构化数据冲突时，以公告原文为准，并说明差异口径。行情、市值或估值类字段若被采集，必须注明查询日期和用途。

## Token Budget And Single-Source Rules

默认采用“单源取数，失败 fallback”，而不是多源并行交叉验证。

| 数据类型 | 默认数据源 | fallback 条件 |
| --- | --- | --- |
| 三表/财务指标 | `$tushare` 或项目既定结构化源二选一 | 接口失败、权限不可访问、关键字段缺失、字段明显异常 |
| 行情/市值/股本 | `$tushare` `daily_basic` 或等价最新市场快照 | 数据不可访问、停牌/复权口径异常、缺市值/股本 |
| 公告/年报/投关 | 巨潮/CNINFO 或交易所原文 | 原文不可访问、PDF 损坏、缺附件 |
| 一致预期/研报预测 | 东方财富研报列表或项目既定一致预期源 | 近端预测缺失或字段不全 |
| 题材/市场热词 | `$wencai-query` | 只在需要市场认知差、题材扩散或行情排名时使用 |
| Gemini/Google Search | 缺口型二次线索发现 | 只在常规来源覆盖不了关键变量时运行；只进入 evidence_queue，永不直接进入 facts_core |

同一字段只允许一个事实来源进入 `facts_core`。需要复核时只记录“口径冲突/待复核”，不要把多个来源的同一字段在多个中间稿里反复铺开。

原始表读取限制：

- 不要对 `*.csv`、问财 JSON/CSV、公告索引和研报列表做宽泛 `rg`。长行命中会造成上下文爆炸。
- 读取原始 CSV 前，先用脚本、`head`、`cut`、`csvcut` 或小型解析代码抽取 10-30 行核心字段。
- 中间稿只引用抽取后的窄表和 `Fact-ID`，不要粘贴完整三表、完整研报列表或完整公告列表。
- 保存原始 CSV 可以，但默认不再读入上下文；只有字段缺失或异常排查时才打开原始表。

## 市场变量分层

底盘型成长股常由“旧业务托底 + 新业务弹性”共同重定价。公告事实确定底线，市场口径识别弹性变量；关键变量要进入主线、利润斜率和跟踪体系，同时标明证据等级。证据边界校准应在主线和利润传导写清后统一执行，不应在采集阶段把券商、行业报告或可交叉验证的产业链线索提前删除。

| 类型 | 例子 | 写法 |
| --- | --- | --- |
| 公告事实 | 财报利润、分业务收入、订单公告、募投产能、管理层明确披露 | 可支撑主结论 |
| 结构化数据 | 股价、市值、财务指标、现金流、应收存货、历史行情 | 支撑财务质量、市场阶段和接力输入 |
| 市场/券商口径 | 客户份额、产品 ASP、毛利率、订单节奏、产业链调研 | 作为影响因子进入主线和情景变量 |
| 行业空间/份额口径 | TAM/SAM/SOM、渗透率、价格/成本/效率曲线、市场份额 | TAM 做想象空间，SAM 约束方向，SOM 才进入收入/利润桥 |
| 合理推断/情景假设 | 根据产能、客户、行业空间和费用率推演的利润斜率 | 标注假设，写清成立/不成立的逻辑降级 |
| 弱线索/传闻 | 无法交叉验证的客户导入、团队变化、排产或份额 | 只作为跟踪变量，不支撑强结论；若出现多源交叉或公开材料印证，可升级 |

对高重要市场变量，必须用段落写清：

- 市场正在交易什么变量。
- 它影响收入、毛利率、费用率、客户粘性、现金流或归母利润中的哪几项。
- 证据属性是什么：公告事实、结构化数据、市场/券商口径、情景假设或弱线索。
- 变量增强时，利润中枢或利润斜率如何变化。
- 变量减弱时，逻辑降级成什么：旧业务修复、普通扩产、概念映射、费用压力，还是主线证伪。
- 哪个公告、财报项目、订单、分业务毛利率、现金流或客户验证能继续确认。

## 单公司深度研究

默认路径：

```text
1. 形成主线假设：市场为什么重新定价，哪些变量决定利润斜率
2. 读取最新年报/季报/业绩预告/重大事项公告
3. 选择一个结构化财务数据源拉财务、现金流、历史行情和市值；估值类字段只进入候选字段、post-report handoff 或后续估值模型，仅在失败或缺字段时 fallback
4. 写 `<标的>_facts_core.md`，把公告事实、财务字段、分业务数据、行情估值快照统一成 Fact-ID
5. 调用 `$wencai-query` 查概念、同行、阶段涨幅、市场排名和交易热词；只保留窄表摘要
6. 搜集券商口径、产业链线索、客户/竞品资料、客户认证线索和公司官网资料，建立市场变量索引
7. 若普通搜索和本地资料仍有关键缺口，且 `GEMINI_API_KEY` 可用，用 `scripts/gemini_google_search.py` 对市场变量、行业空间、客户/竞品、平台复用和利润桥做 1-2 组二次线索发现
8. Gemini/普通搜索结果写入 `<标的>_evidence_queue.md`，核验关键线索后更新 `<标的>_facts_core.md` 和 `<标的>_evidence_index.md`
9. 写 `<标的>_evidence_index.md` 和 `<标的>_market_variables_map.md`
10. 分发给旧业务底盘、第二曲线、行业经济性与蛋糕分配、平台复用、竞争/客户验证链、承接动作、利润桥、跟踪体系和反方审查
11. 写轻量利润中枢、`<标的>_evidence_grading.md`、三表/FCF 与 PEG 估值候选字段/缺口清单，以及 `<标的>_midterm_structure_review.md`
12. 通过中期结构 QA 并完成证据边界校准后写投资人可读报告；终稿不写目标价、目标市值、PE/PEG/SOTP 或买卖建议
13. `final_report` 完成、`skeptic_review` 存在且 `final_report_gate.py` PASS 后，再启动 post-report valuation-handoff stage，写两个正式接力文件
```

## 快速判断

当用户只要快速看一只股票，可以使用轻量模式，并必须说明未启用完整 agentic 流程。

轻量模式至少覆盖：

- 市场真正交易的 3 到 5 个变量。
- 旧业务经营下限。
- 第二成长曲线是否抬收入、抬利润率或抬市场身份。
- 行业经济性、TAM/SAM/SOM、公司份额和利润贡献是否被区分。
- 承接动作是否进入订单、收入、利润或现金流验证。
- 客户验证和竞争格局是否支持第二曲线份额与毛利率假设。
- 利润斜率和最敏感变量。

不能把轻量模式输出包装成完整深度报告。

## 横向比较或选股

当用户要求“找一批”“横向比较”“谁更强”：

```text
`$wencai-query`：先筛候选池、题材口径和市场排名
`$tushare`：补财务质量、现金流、市值和行情指标
公告原文：只抽 Top 候选或争议公司的关键公告
市场线索：按新业务客户、竞争格局、份额、产品价值量和承接动作做差异化比较
```

不要对所有候选公司逐一读年报，先用数据缩小范围。

## 重大事项驱动

当研究重点是重组、定增、可转债、订单、扩产、减持或实控人变化：

```text
公告原文优先
`$tushare` 补市值、行情、现金流和财务承受能力
`$wencai-query` 补市场反应、相关概念和同类标的
券商/产业链口径补充市场正在如何解释该事项
```

最终判断必须区分“事实已公告”“公司预计”“市场推演”和“情景假设”。

## Gemini Google Search 用法

当本地资料、普通 web search、公告和结构化数据无法覆盖关键市场变量时，才可执行：

```bash
python3 /Users/a/.codex/skills/chassis-growth-agentic-research/scripts/gemini_google_search.py "公司名 股票代码 市场变量 行业空间 客户 竞品 券商研报"
```

最多拆 1-2 类最关键查询；不要把下列类型全部常规跑一遍：

- 公司名 + 股票代码 + 新业务/第二曲线 + 券商研报/公告。
- 公司名 + 核心产品 + 客户/竞品/认证/订单。
- 行业关键词 + TAM/SAM/SOM + 渗透率 + 成本/价格/效率。
- 公司名 + 平台复用/扩产/募投/利润率/毛利率。

Gemini 输出必须压缩写入 `evidence_queue` 的“线索来源”，并继续打开原始来源核验；不能把模型总结直接写成正文事实。

## 输出口径

每份报告至少说明：

- 使用了哪些数据路径和 skill 路径。
- 最新数据日期和公告报告期。
- 哪些结论来自公告事实，哪些来自市场口径、合理推断或情景假设。
- 哪些市场变量进入正文、利润斜率和跟踪体系。
- TAM/SAM/SOM、市场占有率、利润贡献和未来天花板如何分层。
- 竞争/客户验证链如何影响第二曲线、承接动作和利润斜率。
- 中期结构 QA 是否通过，若未通过补写了哪些内容。
- 哪些输入只进入候选字段、post-report `dcf_financial_model_handoff`、post-report `peg_valuation_handoff` 或后续估值模型，不进入终稿估值结论。

## Valuation Handoff Order Override

正式估值接力文件不是中期研究产物。单公司深度研究阶段不得提前写正式 `<标的>_dcf_financial_model_handoff.md` 或 `<标的>_peg_valuation_handoff.md`；中期只在利润桥、跟踪体系和证据边界中沉淀三表/FCF 候选字段、PEG 因子、证据等级、情景边界和缺口。

只有同时满足以下条件，才写正式 valuation handoff pair：

```text
<prefix>_final_report.md exists
<prefix>_skeptic_review.md exists
scripts/final_report_gate.py returns PASS
```

gate 通过后启动一个 valuation-handoff subagent 或等价 post-report stage，读取 `final_report`、`skeptic_review`、`profit_bridge`、`tracking_dashboard`、`facts_core` 和必要片段，写两个正式文件：

```text
<prefix>_dcf_financial_model_handoff.md
<prefix>_peg_valuation_handoff.md
```

`dcf_financial_model_handoff` 负责 financial-modeling/DCF-ready 输入；`peg_valuation_handoff` 负责 PEG 因子处理，并必须单独说明每个主要因子如何影响 PEG 系数：提高、降低、封顶、仅允许乐观情景、阻止年份切换或暂不影响。
